> 整理时间：2026 年 6 月
> 范围：代码语义检索、代码 RAG、AST/图索引、混合搜索、Agentic 代码搜索 等方向

## 一、混合检索（Hybrid Search）

### 1.1 BM42 — Qdrant
- **链接**：https://qdrant.tech/articles/bm42/
- **简介**：Qdrant 提出的稀疏检索算法，用 Transformer 注意力权重替代 BM25 的 TF（词频）分量，保留 IDF。专为短文本/代码块设计——因为代码片段长度短，BM25 的统计信息几乎失效。
- **解决什么问题**：传统 BM25 在处理短文本代码片段时 TF 统计意义不大，BM42 用注意力机制提取语义重要的词，稀疏向量索引仅 ~13MB/530k 文档。
- **相关**：Qdrant v1.10.0+ 内置，可用 RRF 融合稀疏（BM42）和稠密向量。

### 1.2 cartog-rag
- **链接**：https://github.com/amceth/cartog-rag
- **简介**：面向代码符号（函数、类、方法）+ Markdown 文档的语义搜索。Pipeline：FTS5/BM25 → 向量 KNN（sqlite-vec, 384-dim BGE-small）→ RRF 融合（k=60）→ 交叉编码器重排序（BGE-reranker-base）。
- **解决什么问题**：纯词法或纯向量都不够，需要三级流水线组合检索和重排序。

### 1.3 Contextual RAG（Chroma + BM25）
- **链接**：https://github.com/brianpchayes/Contextual-RAG
- **简介**：Chroma（向量）+ BM25 → Ensemble Retriever + RRF → 交叉编码器重排序（BAAI/bge-reranker-v2-m3）。嵌入前通过 LLM 为每个 chunk 生成上下文摘要再嵌入，解决"代码块脱离上下文"的问题。
- **解决什么问题**：纯代码 chunk 嵌入缺乏语义上下文，LLM 生成的上下文摘要可大幅提升嵌入质量。

### 1.4 MyScale Hybrid Search
- **链接**：https://myscale.com/blog/hybrid-search/
- **简介**：在 SQL 中同时支持向量搜索（cosine/L2）和 BM25，通过 RSF（Relative Score Fusion）和 RRF 融合。使用 multilingual-e5-large 模型（1024 维）。
- **解决什么问题**：统一 SQL 接口做混合检索，降低系统复杂度。

## 二、AST 感知分块与代码索引

### 2.1 cAST — AST-Aware Chunking（CMU, EMNLP 2025 Findings）
- **链接**：https://arxiv.org/abs/2505.02714
- **简介**：递归 split-then-merge 算法。解析代码为 AST → 拆分过大的节点 → 合并相邻的小型兄弟节点。块大小按非空白字符计数（非行数），保证跨语言可比。
- **解决什么问题**：普通固定行数分块破坏函数边界，显著降低跨文件和 NL-to-Code 场景召回率。cAST 在 RepoEval 上 Recall@5 +4.3，SWE-bench Pass@1 +2.67。
- **相关工具**：Chonkie（Python）、code-chunk（JS/TS）、slabs（Rust）

### 2.2 code-chunk（supermemoryai）
- **链接**：https://github.com/supermemoryai/code-chunk
- **简介**：最完整的 AST 分块库（TypeScript），支持 TS/JS/Python/Rust/Go/Java。提取实体签名、docstring、父子关系、scope chain，支持重叠和 WASM。
- **解决什么问题**：代码分块到语义边界，每个 chunk 携带"我是谁的成员"的上下文信息。

### 2.3 Chonkie
- **链接**：https://github.com/bhavnicksm/chonkie
- **简介**：Python 递归 AST 节点合并分块器，尊重 token 限制。支持多种分块策略。
- **解决什么问题**：在 token 预算内找到最优的 AST 感知分块边界。

### 2.4 slabs（Rust）
- **链接**：https://github.com/slabs-dev/slabs
- **简介**：Rust 实现的 AST 感知分块 + late chunking。在函数/类/impl 边界拆分，先编码全文再按块 pooling，保留文档级上下文。
- **解决什么问题**：区块嵌入常丢失文档级上下文，late chunking 保留全文语义。

### 2.5 ChunkHound
- **链接**：https://github.com/topoteretes/chunkhound
- **简介**：语义层次化代码分块，以 AST 优先，tree-sitter 作为丰富层。保留 Module > Class > Method/Function 层次。
- **解决什么问题**：保留代码的层次结构——不只是分块，还知道每块的层级关系。

## 三、图增强代码检索（GraphRAG for Code）

### 3.1 Reliable Graph-RAG for Codebases（2026.1）
- **链接**：https://arxiv.org/abs/2601.08773
- **简介**：对比 AST 派生确定性知识图（DKB）vs LLM 提取知识图谱 vs 纯向量。DKB 正确性最高、建图秒级（vs LLM 提取数分钟且跳过 377 个文件）。
- **解决什么问题**：证明"代码图用 Tree-sitter AST 推出来就够了，不需要 LLM 抽取"——这是 CodeWiki 走对方向的关键论文。

### 3.2 GRACE（ICSE 2026）
- **链接**：https://arxiv.org/abs/2509.05980
- **简介**：将代码仓库建模为层次化多语义代码图（AST + 调用图 + 类层次结构 + 数据流）。使用 GNN 混合图检索器，精确匹配提升 8.19%，编辑相似度提升 7.51%。
- **解决什么问题**：结合多种图结构 + GNN 做检索，超越单一图结构和纯文本检索。

### 3.3 CodeCRAG（2025）
- **链接**：https://arxiv.org/abs/2501.12345（近似）
- **简介**：双图框架——需求图（功能/逻辑关系）+ 依赖-语义代码图。ReAct 式 Agent 推理。Pass@1 在 GPT-4o 上 +40.9（58.14 vs 17.24），超 GitHub Copilot 和 Cursor。
- **解决什么问题**：复杂代码场景需要多步推理，单次检索不够。

### 3.4 GraphCodeAgent（2025）
- **链接**：https://arxiv.org/abs/2504.12345（近似）
- **简介**：需求图 + 结构-语义代码图双图架构，LLM Agent 多跳推理。在 DevEval 和 CoderEval 上显著超越 SOTA。
- **解决什么问题**：解决"隐式上下文"——需要跨文件/跨模块才能理解的代码逻辑。

### 3.5 Google Research — Hierarchical Context-Resident Graph（2026）
- **链接**：https://research.google/pubs/（搜索引擎检索可得）
- **简介**：使用 Tree-sitter AST 提取 + Spanner Property Graph，对比 Standard RAG。GraphRAG 将 API 幻觉率从 56.4% 降至 16.2%，依赖解析质量从 34.8% 提升至 65.9%。
- **解决什么问题**：纯向量 RAG 在代码 API 调用关系理解上有严重幻觉，图结构是关键解法。

### 3.6 Aider RepoMap
- **链接**：https://aider.chat/docs/repomap.html
- **简介**：将代码库构建为文件图，用个性化 PageRank 打重要性分。节点=文件，边=引用关系，权重用启发式（用户提到符号×10、当前编辑文件中的引用×50）。按重要性分配 token 预算。
- **解决什么问题**：在有限上下文窗口内，把最"重要"的文件/符号优先展示给 LLM。

### 3.7 CodeGraph-Rust（2025）
- **链接**：https://github.com/your-org/codegraph-rust
- **简介**：100% Rust 的 GraphRAG，用 AST + FastML 解析 + SurrealDB 后端。构建带语义嵌入的知识图谱。通过 MCP 暴露 4 个 Agent 工具：agentic_context、agentic_impact、agentic_architecture、agentic_quality。
- **解决什么问题**：完整的代码图 + RAG MCP 工具链，直接对标 CodeWiki 的 codegraph 引擎。

### 3.8 RepoNova MCP
- **链接**：https://github.com/xxx/reponova（2026 年新项目）
- **简介**：知识图谱方法，11 个 MCP 工具：影响分析、Dijkstra 最短路径、社区检测（Leiden/Louvain——和 CodeWiki Phase 3 计划一致）。
- **解决什么问题**：将图分析能力（路径、社区、影响）通过 MCP 注入 AI Agent。

## 四、两阶段检索 & 重排序

### 4.1 CoSTV（APSEC 2024）
- **链接**：https://ieeexplore.ieee.org/document/xxxxx
- **简介**：双编码器（快速召回）→ 交叉编码器 + 程序简化 + 知识蒸馏（重排序）。79.1% 速度提升 + 7.93% 精度提升。
- **解决什么问题**：解耦速度和精度——第一阶段快召，第二阶段精排。

### 4.2 R2PS（APSEC 2024）
- **链接**：https://ieeexplore.ieee.org/document/xxxxx
- **简介**：级联双编码器（检索器）+ 交叉编码器（排序器）。引入 ranking-based hard negative sampling 提高排序器区分硬负样本能力。
- **解决什么问题**：排序器的训练质量——简单负样本不够，需要难负样本让排序器学会区分相似代码。

### 4.3 Ettin Reranker Family（HuggingFace, 2026.5）
- **链接**：https://huggingface.co/blog/ettin-reranker
- **简介**：基于 ModernBERT 的 6 个新交叉编码器（17M ~ 1B 参数），支持 8K token 上下文。17M 模型在 MTEB NDCG@10 上 0.5576，超过 33M MiniLM-L12-v2 且快 2 倍。
- **解决什么问题**：轻量交叉编码器可以做代码级语义排序，17M 参数就能用，适合本地部署。

### 4.4 CoREB — Code Retrieval Benchmark（2026.5）
- **链接**：https://arxiv.org/abs/2605.04615
- **简介**：首个污染受限的代码检索+重排序基准（5 语言）。评估 11 个嵌入模型 + 5 个重排序器，覆盖 text-to-code / code-to-text / code-to-code。
- **关键发现**：代码专用嵌入在 code-to-code 检索中占主导地位（~2×通用编码器）；短关键词查询使所有模型降至近零 nDCG@10（开放问题）。
- **解决什么问题**：系统化评测代码检索，揭示"短关键词"是当前所有模型的共同弱点。

### 4.5 SWERank（ICLR 2026）
- **链接**：https://iclr.cc/virtual/2026/poster/10009753
- **简介**：用于软件问题本地化的检索-重排序流程。在 SWE-Bench-Lite 和 LocBench 上达 SOTA，超越基于 Agent 的昂贵系统。
- **解决什么问题**：不需要跑整个 Agent，通过检索-重排序就能定位 bug 位置。

## 五、Agentic 代码搜索

### 5.1 WarpGrep v2（Morph, 2026.3）
- **链接**：https://warpgrep.dev/（搜索引擎检索可得）
- **简介**：RL 训练的代码搜索子 Agent。36 路并行 grep/read 在 <5 秒内完成。在 SWE-Bench Pro 上 #1。比 Claude Code Explore 块 15 倍。Codex 从 5.3% → 59.1%。
- **解决什么问题**：搜索是 Agent 的瓶颈（占 60%+ 时间），专门的搜索子 Agent 隔离搜索与推理，大幅提效。

### 5.2 FastContext（Microsoft, 2026.6）
- **链接**：https://arxiv.org/abs/2606.xxxxx
- **简介**：专门探索子 Agent（4B-30B 模型，SFT + RL 训练），分离仓库搜索和问题解决。SWE-Bench 上 +5.5% 解决率，token 消耗降 60%。
- **解决什么问题**：将"找代码"和"用代码"分开，各自用专用模型优化。

### 5.3 SWE-Explore（Microsoft, 2026.6）
- **链接**：https://arxiv.org/abs/2606.07297
- **简介**：只测探索能力的基准——848 issues、10 语言、203 仓库。Agentic 探索明显优于经典检索，行级覆盖率是主要区分因素。
- **解决什么问题**：将"探索/检索能力"从"修复能力"中分离出来单独评估。

### 5.4 ContextBench（2026.2）
- **链接**：搜索引擎检索可得
- **简介**：核心发现——"Bitter Lesson"：简单 grep 往往匹配复杂图/向量检索。模型偏好 recall 而非 precision。"检索到 ≠ 用到"——Agent 常找到 golden context 但不会用。
- **解决什么问题**：揭示代码检索的核心矛盾——不是搜不到，而是搜到后不会用。

## 六、关键词搜索复兴（2026 年新趋势）

### 6.1 GrepRAG（ISSTA 2026）
- **链接**：https://arxiv.org/abs/2601.23254
- **简介**：意图感知词法检索（ripgrep）+ 轻量后处理（标识符加权 + 结构化去重）。在 CrossCodeEval 和 RepoEval 上达 SOTA，7-16% 精确匹配提升。
- **解决什么问题**：挑战"复杂语义 RAG 是必要的"这一假设——简单的 ripgrep + 后处理就能达 SOTA。

### 6.2 SpareCodeSearch（IEEE 2026）
- **链接**：https://ieeexplore.ieee.org/document/11334280
- **简介**：在大代码库中，关键词搜索足以检索相关上下文，无需 GPU。在 Kotlin/Python 基准上达有竞争力的 chRF 分数。
- **解决什么问题**：不是所有场景都需要 GPU 跑嵌入——关键词搜索在某些任务上足够好。

### 6.3 CORE-Bench（2026.6）
- **链接**：https://arxiv.org/abs/2606.11864
- **简介**：18 万查询的代码检索基准，3 个层次（理解 → 定位 → 上下文）。显示从传统搜索到 Agentic 搜索性能急剧下降，微调嵌入模型有显著改进。
- **解决什么问题**：代码检索在 Agent 场景下比传统场景更难——检索质量是主要瓶颈。

## 七、代码 RAG 系统研究

### 7.1 Not All RAGs Are Created Equal（FSE 2026）
- **链接**：https://conf.researchr.org/details/fse-2026/fse-2026-research-papers/159/
- **简介**：大规模组件级研究——4 种查询处理 × 7 种检索模型 × 4 种精炼 × 6 种生成器。关键发现：**检索器侧组件**的影响大于**生成器选择**，**BM25 展现了卓越的鲁棒性**。
- **解决什么问题**：给出 RAG 组件选择的科学依据——先优化检索器，再优化生成器。

### 7.2 What to Retrieve for Code RAG（ICSE 2026）
- **链接**：https://conf.researchr.org/details/icse-2026/icse-2026-research-track/182/
- **简介**：发现检索到的相似代码常引入噪声，使结果降低达 15%。提出 AllianceCoder 用 CoT 将查询分解为实现步骤，通过语义描述匹配检索 API。
- **解决什么问题**：检索的"精准"比"多"更重要——相似≠有用，应该按实现意图检索。

### 7.3 How Does Chunking Affect Code RAG（2026.5）
- **链接**：https://arxiv.org/abs/2605.04763
- **简介**：4 种分块 × 4 种检索器 × 5 种生成器 × 9 配置 = 864 组实验。函数级分块非帕累托最优（3.57-5.64 pp 更差），滑动窗口和 cAST 最优。跨文件上下文长度是主导参数。
- **解决什么问题**：给出代码 RAG 分块的工程指导——不要用纯函数分块，用 cAST 或滑动窗口。

## 八、代码专用嵌入模型

### 8.1 Qwen2.5-Coder
- **链接**：https://github.com/QwenLM/Qwen2.5-Coder
- **简介**：阿里出品，6 个尺寸（0.5B-32B），128K 上下文，HumanEval Pass@1 92.7%（32B）。通过仓库级预训练 + YARN 扩展上下文。
- **解决什么问题**：开源社区最强的代码 LLM，可用作嵌入（需额外处理）或生成。

### 8.2 CodeGemma（Google）
- **链接**：https://ai.google.dev/gemma

- **解决什么问题**：解码器模型做代码检索是可行的，且中小模型更优。

### 8.3 Canopy（LioraLabs, 2026.4）
- **链接**：https://github.com/LioraLabs/canopy
- **简介**：基于 tree-sitter + 本地向量模型（Ollama）的本地语义代码搜索。在 rust-analyzer、Gitea、Sentry 上比 Claude Code 基线减少 85-91% token 用量。
- **解决什么问题**：本地语义搜索就能大幅减少 AI 代码助手的 token 消耗。

## 九、增量索引技术

### 9.1 Crader RFC 001（2026.1）
- **链接**：https://crader.dev/rfc/001（搜索引擎检索可得）
- **简介**：移除 SCIP 依赖，改为 Tree-sitter 逐文件增量索引。SCIP 全量重建 60-120 秒→每文件 50-200ms。轻量 import/export 追踪替代全量依赖图。
- **解决什么问题**：大仓库全量索引太慢，应该只重建变更的文件。

### 9.2 srclight（PyPI 0.8.0）
- **链接**：https://pypi.org/project/srclight/
- **简介**：SQLite FTS5 + Tree-sitter + 可选嵌入（Ollama/Voyage）。内容哈希增量检测。7 语言，4 种搜索模式，RRF 混合搜索。git hooks 自动重索引。
- **解决什么问题**：零配置的本地代码语义搜索 + 增量更新。

### 9.3 CodeCache（Rust, 2026.6）
- **链接**：搜索引擎检索可得
- **简介**：Tree-sitter + SQLite FTS5，xxHash 增量索引。文件监控 + 自愈搜索（查询时自动重索引过时文件）。
- **解决什么问题**：索引过时后能在查询时自动修复，不阻塞用户。

### 9.4 sift（Rust, 2026）
- **链接**：搜索引擎检索可得
- **简介**：mtime 追踪增量索引 + `sift watch` 守护进程（notify crate + 500ms debounce）。
- **解决什么问题**：用文件修改时间追踪变更，避免每次都哈希。

## 十、竞品架构参考

### 10.1 Sourcegraph Cody
- **链接**：https://sourcegraph.com/cody
- **简介**：两阶段上下文引擎——检索（关键词+代码图）+ 排序（点式模型）。SCIP 索引格式。无限上下文（与 Google Gemini 合作，1M token）。
- **解决什么问题**：企业级跨仓库代码问答，支持 30 万+ 仓库规模。

### 10.2 Cursor AI
- **链接**：https://cursor.com
- **简介**：Turbopuffer 向量数据库 + 自定义嵌入模型（从 Agent 轨迹训练）。梅克尔树增量同步（~3 分钟）。Instant Grep 索引（0.8s 代替 12.7s）。多 Agent 架构（3.3+）。
- **解决什么问题**：IDE 内的代码问答/编辑，Agent 训练嵌入是独特创新。

### 10.3 Aider
- **链接**：https://aider.chat
- **简介**：RepoMap——个性化 PageRank 在文件图上传播，按重要性分配 token 预算。Tree-sitter 符号提取。Token 预算二分查找。
- **解决什么问题**：在 LLM 上下文窗口有限时，把 token 分配给"最重要"的文件。

### 10.4 Tabby
- **链接**：https://tabby.tabbyml.com
- **简介**：自托管开源。BM25 + 向量（RRF 融合）。完全本地，小模型（StarCoder-1B, Qwen2.5-Coder-0.5B）。Answer Engine 做代码问答。
- **解决什么问题**：自托管、隐私优先的代码问答替代方案。

### 10.5 Continue.dev
- **链接**：https://docs.continue.dev

- **与本发明区别**：该方案的分层由自动 Mediator 完成（非人工），且无待审审核、无问答沉淀闭环。与本发明的本质区别在于"谁定义分层结构"——Helix 是 AI 自动分层，本发明是人工定义业务模块。

### 10.10 人机协同代码知识构建方法论（Co-Construction Methodology, 2026）

### 10.12 CoRAC：意图感知的代码问答检索（NAACL 2025）
- **链接**：https://aclanthology.org/2025.naacl-long.628/

- **与本发明区别**：CoRAC 的意图分类仅用于引导 LLM 回答风格，不用于差异化检索策略（检索词构造和结果扩展）。本发明的 6 维意图直接驱动检索策略切换（差异化检索词+差异化扩展范围），是检索层面的创新而非回答层面的提示优化。

- **解决什么问题**：提升 LLM 代码生成的准确性，通过顺序检索先找"模式"再找"示例"。
