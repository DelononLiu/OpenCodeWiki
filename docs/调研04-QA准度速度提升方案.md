# OpenCodeWiki QA 准度与速度提升方案

> 2026 年 6 月 · 目标定位：**团队智能代码知识库（≤20 仓库）**
> 关键结论：codegraph 不动、sqlite-vec 做向量、增量索引必需、多库原生支持
> 进度度量：迭代（非时间）· 评测方式：SWE-QA 基准 + 自建问题

---

## 一、当前瓶颈

2026 年 FSE/ICSE 研究验证的瓶颈判断：

```
瓶颈不在图，在检索

当前:
  问题 → FTS5 符号搜索 → 直接送 LLM
          ↑ 无语义匹配     ↑ 含噪声

应有:
  问题 → FTS5 + 向量(RRF) → 交叉编码器重排序 → LLM
          ↑ 语义+词法全      ↑ 去噪声
```

| 瓶颈 | 影响 | 优先级 |
|------|------|--------|
| 仅有 FTS5 词法搜，无语义 | 问"支付"搜不到 refund | **P0** |
| 结果无重排序直接喂 LLM | 噪声占 token、回答跑偏 | **P0** |
| 增量索引需手动 | 搜到的代码过时 | **P0** |
| 多库搜索无路由全部并行 | 跨库问题噪声多 | **P1** |
| 符号级索引缺整体上下文 | 类/模块级问题要拼凑 | **P1** |

---

## 二、改动原则

1. **不动 codegraph 引擎**——向量检索等增量能力加在上层
2. **sqlite-vec**——与现有 SQLite 共存，零基础设施
3. **评测先行**——用 SWE-QA 做基座，每迭代对比基线

---

## 三、评测方案

### 评测集：两级规模

| 级别 | 规模 | 用途 |
|------|------|------|
| **tiny** | 10 题 | 开发期间快速验证，每次改完跑一轮 |
| **full** | SWE-QA 720 题 + 自定义 | 完整回归，迭代发布前跑 |

**tiny 集**：从 SWE-QA 选 5 题（不同仓库）+ 自己仓库选 5 题。
**full 集**：全部 SWE-QA 题，从答案中提取 `file:line` 引用做 golden context + 自定义补充（10-20 题）。

| 指标 | tiny | full |
|------|------|------|
| 跑一次耗时 | ~30 秒 | ~5 分钟 |
| 置信度 | 快速反馈 | 完整回归 |

### 评测输出示例

```
$ npm run eval:qa tiny

===== tiny (10 题) =====
Recall@5:   0.70  (基线: 0.50)  ▲ +20%
MRR:        0.65  (基线: 0.48)  ▲ +17%
平均检索延迟: 180ms (基线: 45ms)
```

```
$ npm run eval:qa full

===== full (420 题) =====
Recall@5:   0.72  (基线: 0.58)  ▲ +24%
MRR:        0.68  (基线: 0.55)  ▲ +24%
回答正确率:  67%   (基线: 55%)   ▲ +12%
```

## 四、迭代计划

### Iteration 0：评测基座

**目标**：SWE-QA 适配 + 基线数据

**交付**：
- `scripts/eval/` 目录：
  - `swiqa_adapter.mjs` — 解析 SWE-QA 答案提取 golden context
  - `custom.json` — 自定义问题（10-20 题）
  - `golden.json` — 合并后的完整评测集
  - `runner.mjs` — 逐题跑检索 + 统计指标
  - `report.mjs` — 输出对比报告
- 跑 `npm run eval:qa baseline` 输出当前系统基线

**补充说明**：SWE-QA 的 15 个仓库需要 `clone_repos.sh` 拉取（已有脚本），然后跑 `npm run index` 让 codegraph 索引，最后跑评测。

---

### Iteration 1：混合检索

**一句话**：让 FTS5 搜不到的语义内容也能被搜到。

**改动**：
- `scripts/setup-repo.mjs` — codegraph 索引完触发嵌入生成（**C2LLM-0.5B, 512 维**）
- `src/server/qa-resolver.ts` — 新增 HybridSearchService：
  - `codegraph_search` 走 FTS5 → top-20
  - `vector_search` 走 sqlite-vec → top-20
  - RRF 融合 → top-20 → 交下游

**向量库**：`~/.opencodewiki/vectors/<repo>.vec.db`（与 codegraph.db 平级，不动 codegraph）

**验证**：`npm run eval:qa iter1` 对比 Iteration 0，Recall@5 提升 ≥10%

---

### Iteration 2：重排序 + 模板精简

**2a. 交叉编码器重排序**
- 集成 **Ettin-reranker-32m-v1**（32M 参数，8K 上下文，NDCG@10=0.5779）
- 插入位置：RRF 融合后、expandWithContext 前
- 流程：RRF top-20 → reranker → top-5 → expand

**2b. 模板动态选择**
- 当前：5 模板 + 6 领域流全注入 → ~2K token 浪费
- 改为：intent + domain 组合 → 1 个模板 + 对应领域流

**验证**：回答正确率提升、system prompt 减少 ~600 tokens

---

### Iteration 3：增量索引 + 多库路由

**3a. 增量索引**
- `src/server/codegraph-bridge.ts` — 新增文件变更监听端点
- `scripts/reindex.mjs` — 支持 `--watch` 模式 + git post-merge hook
- 变更文件 → xxHash 比对 → 仅重解析变更文件 → 更新嵌入

**3b. 多库路由**
- 问题含 `@repoName` → 直达目标仓库
- 无指定 → 用仓库 NL 摘要（`wiki/overview.md`）+ 问题相似度 → top-3 候选
- 置信度 < 0.5 回退全量并行

**验证**：修改代码后不用手动 reindex 可搜；跨库回答噪声减少

---

### Iteration 4：上下文增强 + 查询优化

**4a. 类/文件/模块级 chunk**
- 在符号级索引之外，补充粗粒度 chunk（类级/文件级/模块级）
- 每个 chunk 独立嵌入，携带层级元数据
- 检索时符号级 + chunk 级并行搜，RRF 融合

**4b. 查询改写**
- 驼峰/蛇形拆词（UserAuthService → user auth service）
- 词形还原（caching → cache）
- 搜不到时用小模型同义改写

**4c. 图传播重打分**
- 从命中符号沿调用图 2-3 跳传播
- 被多路径引用的中间函数获得传播加分
- 主要用于 `what-impact` 意图的调用链补全

**验证**：`npm run eval:qa iter4` 全量指标环比提升

## 五、不做清单

| 项目 | 原因 |
|------|------|
| Agentic 搜索回路 | 搜索质量够好时不需要——一次搜准才是根本解法 |
| 替换 codegraph | 2026 论文确认 AST 派生图正确性最高 |
| LanceDB / Qdrant | sqlite-vec 在现行架构里够用 |
| 模板 JSON schema | 当前 prompt 约束已够 |
| Skills/Playbook 系统 | 硬编码已覆盖主要场景，可控到 Iteration 4 末尾 |
| BERTScore 回答评测 | 用 all-MiniLM-L6-v2 即可，当前 ROUGE-L 够用 |
| LLM-as-judge 回答评测 | 调 LLM 自身打分，更贴近人类判断但花钱 |

## 六、技术选型

| 组件 | 选择 | 理由 |
|------|------|------|
| 评测基座 | **SWE-QA**（720 题）+ 自定义补充 | 最贴近的公开代码问答基准 |
| 向量库 | **node:sqlite + 纯 JS 余弦搜索** | 零 native build（vs sqlite-vec 需编译扩展），暴力 50k×384d ~5ms |
| 嵌入模型 | **C2LLM-0.5B**（512 维） | 2026 CoREB 基准 #4，代码专用，0.5B 本地 CPU 可跑 |
| 重排序器（生产） | **Qwen3-Reranker-4B** | 2026 年代码 rerank SOTA，Apache 2.0 |
| 重排序器（开发） | **Ettin-reranker-32m-v1**（32M） | CPU 可跑，替代生产模型做管道调试 |
| 嵌入运行时 | **@xenova/transformers** | 纯 JS ONNX 推理，零 native 依赖 |
| 向量存储 | `~/.opencodewiki/vectors/<repo>.vec.db`（node:sqlite） | 与 codegraph.db 平级，不侵入 codegraph |
| 融合方式 | **RRF (k=60)** | 简单有效，2026 生产标配 |

### 模型部署策略：生产 API + 开发本地

| 组件 | 生产 | 开发调试（本地 CPU）| 切换方式 |
|------|------|-------------------|---------|
| 嵌入 | API 调用 C2LLM-0.5B | **all-MiniLM-L6-v2**（384d, 80MB）| `NODE_ENV=development` |
| 重排序 | API 调用 Qwen3-Reranker-4B | **Ettin-reranker-32m-v1**（32M） | `NODE_ENV=development` |

开发版和生产版的管道逻辑完全一致，只有模型质量差异。开发阶段跑 `eval:qa` 分数低是正常的，上生产切 API 后跳一个大台阶，反而能验证模型升级的效果。
