# 技术交底书

## 一、发明名称

一种基于问答交互的代码知识结构化沉淀与复用方法及系统

---

## 二、技术领域

本发明涉及软件开发工具技术领域，具体涉及一种基于大语言模型（LLM）的代码知识问答、跨仓库检索与问答知识结构化沉淀的方法及系统。

---

## 三、背景技术

### 3.1 现有技术概述

随着大语言模型（LLM）技术的快速发展，基于 LLM 的代码理解与问答工具（如 Sourcegraph Cody、Cursor AI、GitHub Copilot Chat 等）已成为软件开发中的常用辅助手段。这些工具能够根据开发者的自然语言提问，在代码仓库中检索相关信息并生成回答。

然而，现有技术存在以下不足：

**（1）问答结果孤立，无法沉淀为团队知识。** 每次问答都是独立的会话，问答过程产生的知识无法被自动编号、引用、关联和校准。团队中不同成员可能反复提出相同问题，每次都需要重新执行检索和生成，无法复用已确认的正确答案。

**（2）检索策略单一，无法适应不同意图的代码问题。** 现有方案对"这个函数是做什么的"、"这个函数定义在哪里"、"这个函数怎么使用"、"为什么报这个错误"等不同类型的代码问题通常采用相同的检索策略（统一的 BM25 或向量检索），未针对不同意图优化检索词构造和结果扩展方式，检索精度受限。

**（3）未利用代码特有的调用关系优化检索排序。** 仅使用传统的文本匹配或语义匹配排序，忽略了代码中函数、模块之间的调用关系对相关性排序的提升作用——被大量调用的核心函数应当比同等文本匹配度的一般函数有更高的排序优先级。

**标题：** 一种基于大模型驱动代码反向生成设计书的方法及系统  
**申请人：** 中国实体（2025 年授权）

**核心内容：**

**申请人：** Open Weaver Inc.（现属 Google LLC，2021 年申请）

**标题：** CodeWiki: A Repository-level Multi-language Documentation Generation Framework  
**作者：** FPT Software AI Center（2025 年 11 月发布）

**存在的缺陷：**
该方法仍然是一次性的文档生成流程，未涉及"从问答交互中沉淀知识"的机制。多智能体递归处理的成本较高（每次生成均需多次 LLM 调用），且生成的文档与仓库绑定，不支持跨仓库搜索和引用。此外，该方法缺乏对问答结果的版本化校准和团队协作校准能力。

---

### 3.3 现有技术总结

综上所述，现有技术存在三个主要技术空白：

1. **缺少从问答交互到结构化知识沉淀的闭环机制**——问答知识无法自动编号、关联、校准和复用；
2. **缺少意图自适应的代码检索策略**——不同类型问题采用相同的检索方式，精度受限；
3. **缺少利用代码调用关系图优化搜索排序的手段**——文本匹配和语义匹配之外的代码特有的排序信号被忽略。

本发明正是针对上述三个技术空白提出的完整解决方案。

## 四、发明内容

### 4.1 要解决的技术问题

本发明要解决的根本技术问题是：**如何将代码问答交互中产生的知识，以结构化、可引用、可校准、可自动归档的方式沉淀下来，使其从"一次性交互结果"转变为"团队可复用的知识资产"。**

在此基础上，本发明还进一步解决以下从属问题：

1. 如何根据用户提问的意图自适应选择检索策略，提高代码检索的精度；
2. 如何利用代码调用关系图优化搜索结果排序，使核心函数获得更高的排位；
3. 如何在多仓库场景下自动判定搜索范围。

### 4.2 技术方案概述

本发明提出了一种以**问答知识结构化沉淀**为核心、以**意图自适应检索**和**调用图传播重打分**为增强手段的代码知识问答方法及系统。

**核心机制——#Q 知识沉淀子系统：**
每次问答交互自动分配全局唯一的递增编号（#Q 编号），支持条目间的双向关联链接、版本化人工校准，以及通过 `#Q+编号` 语法在后续问答中引用历史条目；经校准的问答条目自动沉淀为团队知识资产，并流入 Wiki 知识库形成可浏览的结构化文档。

**增强机制一——意图自适应检索：**
通过 LLM 分类器将自然语言问题映射到 6 种预定义代码意图之一，每种意图对应不同的检索词构造策略和结果扩展策略。

**增强机制二——调用图传播重打分：**
在混合检索（FTS5 + 向量搜索 + RRF 融合）之后，沿代码调用关系图的边对 Top-N 结果进行分数传播和衰减重打分。

**增强机制三——跨仓库范围判定：**
支持 `@repoName` 指定仓库、`@cross` 跨仓库搜索，以及基于自然语言的自动范围分类。

其中，核心机制独立构成完整的技术方案，各增强机制可分别或组合地与核心机制配合实施。

## 五、技术方案

### 5.1 #Q 问答知识沉淀系统

#### 5.1.1 数据结构设计

系统维护两个核心数据库表：

**问答条目表（qa_entries）** 包含以下字段：自增编号 qid（从 10000 开始全局递增）、会话标识 sessionId、关联仓库 repo、所属模块 module、原始问题 question、LLM 回答 answer、查询模式 mode（轻量/深度）、领域分类 domain、状态 status（活跃/归档）、父条目 parent_qid、关联条目编号列表 related_qids、标签 tags、引用来源 sources、访问计数 visit_count、时间戳。

**校准答案表（calibrated_answers）** 包含以下字段：关联问答条目 ID qaEntryId、校准答案内容 answer、校准人标识 calibrator、校准理由 reason、版本号 version（首次为 1，每次校准递增）、时间戳。

#### 5.1.2 自增编号机制

系统在创建新问答条目时，通过查询当前最大 qid 值并加 1 的方式生成新的 #Q 编号。编号从 10000 起始，保证每个条目拥有全局唯一的、人类可读且递增的标识。该编号在后续问答中可通过 `#Q123` 语法引用。

#### 5.1.3 双向关联链接机制

`linkEntries(sourceQid, targetQid)` 方法在两个问答条目之间建立双向对称关联：

```
步骤 1：获取源条目 source 和目标条目 target
步骤 2：将 targetQid 添加到 source.relatedQids（如已存在则跳过）
步骤 3：将 sourceQid 添加到 target.relatedQids（如已存在则跳过）
步骤 4：同时更新两条记录的时间戳
```

该机制确保关联关系始终是对称的，任意一端建立的关联在另一端自动可见。

#### 5.1.4 版本化校准机制

`upsertCalibratedAnswer()` 方法支持对任意问答条目进行人工校准：

```
首次校准：创建校准记录，version = 1，记录校准人和校准理由
再次校准：version 自动 +1，保留首次校准的 createdAt，更新 updatedAt
```

每次校准都会递增版本号，使得校准历史可追溯。

#### 5.1.5 #Q 引用语法与自动解析

在后续问答输入中，系统自动通过正则表达式 `/#Q(\d+)/g` 解析 `#Q123` 格式的引用：

```
步骤 1：检测问题中的 #Q 引用
步骤 2：通过 getEntryByQid() 获取被引用的条目
步骤 3：自动调用 linkEntries() 建立双向链接
步骤 4：将被引用条目的上下文注入 LLM 提示词
```

### 5.2 6 维意图自适应代码检索方法

#### 5.2.1 意图定义

系统预定义 6 种面向代码问答的意图：

| 意图标识 | 含义 | 典型问题示例 |
|---------|------|-------------|
| what-is | 功能解释 | "这个函数是做什么的"、"解释一下这段代码" |
| where-is | 定位 | "这个类定义在哪里"、"实现在哪" |
| how-to | 用法示例 | "怎么用这个 API"、"如何调用" |
| why-error | 排错分析 | "为什么报这个错"、"分析这个堆栈" |
| what-structure | 架构分析 | "整体架构是什么"、"模块关系图" |
| what-impact | 影响分析 | "改了 A 会影响谁"、"调用链分析" |

#### 5.2.2 两阶段意图分类器

系统采用主备两阶段分类器：

**第一阶段（主分类器）：LLM 分类**

通过向 LLM 发送带 JSON Schema 的 system prompt，要求返回结构化的分类结果：

```
System Prompt: "你是一个代码意图分析师。根据问题和可用仓库，返回 JSON 格式：
  意图（intent）：what-is | where-is | how-to | why-error | what-structure | what-impact
  范围（scope）：single | cross-compare | cross-call | global-search
  返回格式：{"intent":"what-is","scope":"single","reasoning":"..."}"
```

设置 temperature=0 保证分类的确定性。response_format 设置为 json_object 确保输出可解析。

**第二阶段（降级分类器）：规则匹配 + 置信度打分**

当 LLM 分类不可用（网络异常、API 不可达等）时，使用预定义的正则规则集合进行意图分类。每条规则包含正则模式和优先级，最终选择匹配规则中优先级最高的意图作为输出。同时计算置信度分数：

```
置信度 = 匹配比例 × 0.5 + 优先级因子 × 0.5
其中优先级因子 = min(1, (最高优先级 + 1) / 6)
```

置信度低于 0.3 时默认使用 what-is 意图。

#### 5.2.3 差异化检索策略

根据分类得到的意图，系统从预定义的检索策略表中选择对应的策略执行。每种策略包含检索词构造规则和结果扩展规则两部分：

**（a）what-is（功能解释）策略：**
- 检索词构造：使用提取的符号名 + 上下文关键词（当符号名为空时使用去除代码关键词后的自然语言词）
- 结果扩展：对 Top-N 结果调用 codegraph_context 工具展开完整定义内容

**（b）where-is（定位）策略：**
- 检索词构造：仅使用精确符号名，不添加任何扩展词
- 结果扩展：最小化扩展路径，仅返回定位所需的最小上下文

**（c）how-to（用法示例）策略：**
- 检索词构造：使用符号名 + 预定义提示词组合（config、example、usage、setup、init）
- 结果扩展：对 Top-N 结果同时展开调用者和被调用者信息，展示调用链上下文

**（d）why-error（排错）策略：**
- 检索词构造：两阶段构造——第一阶段提取大写错误码（如 EACCES、TIMEOUT）和错误值模式，第二阶段提取常规符号名
- 搜索执行：第一阶段对错误码进行文本 grep，第二阶段对符号名进行 codegraph 搜索
- 结果扩展：合并两阶段结果后展开上下文

**（e）what-structure（架构）策略：**
- 检索词构造：使用符号名 + 类型提示词（module、interface、type、enum）+ 通用结构词（index、main、README、package）
- 结果扩展：展开结构定义和类型层次

**（f）what-impact（影响分析）策略：**
- 检索词构造：使用符号名 + usage、reference 等影响相关词
- 结果扩展：对 Top-1 结果展开其全部调用者和影响范围（BFS，深度 2）

#### 5.2.4 搜索范围分类

系统同时分类搜索范围，决定在哪些仓库中执行检索：

```
规则 1：问题中出现多个仓库名 → cross-compare（跨库对比）
规则 2：问题中出现单个仓库名 → single（单库）
规则 3：影响/调用链关键词 → impact / cross-call
规则 4：对比类关键词 → cross-compare
规则 5：跨库引用关键词 → cross-call
规则 6：模糊搜索关键词 → global-search
规则 7：以上均不匹配 → single（默认）
```

### 5.3 代码调用关系图的构建

本发明所依赖的代码调用关系图通过以下方式构建：

从源代码中解析出函数定义、函数调用、类定义、类继承、模块导入等代码结构信息；将每个结构单元抽象为一个图节点，将结构间的引用关系（如函数A调用了函数B、类C继承了类D、模块E导入了模块F）抽象为图的有向边；经过跨文件符号解析后，将分散在不同文件中的同名符号关联为同一节点，最终构建为一个覆盖全仓库的统一有向图——节点代表代码符号，边代表符号间的调用/引用/继承/导入关系。

该图以邻接表形式持久化存储，支持高效的调用链查询和影响范围遍历。

### 5.4 调用图传播重打分方法

本发明的混合搜索管道采用以下阶段式架构：

```
阶段 1：FTS5 全文搜索（codegraph_search）
阶段 2：向量语义搜索（cosine 相似度）
阶段 3：RRF 融合（融合阶段 1 和阶段 2，k=60）
阶段 4：调用图传播重打分
阶段 5（可选）：交叉编码器重排序（Python 子进程）
```

其中阶段 4 的调用图传播重打分方法如下：

```
输入：RRF 融合后的排序结果列表 R
参数：Top-N = 3（传播种子数），衰减系数 = 0.15

步骤 1：取 R 的前 N 个结果作为传播种子 S
步骤 2：对每个种子 s ∈ S：
  a. 查询代码调用关系图，获取 s 的直接调用者和被调用者
  b. 对每个相邻节点，增加分数 = s.分数 × 衰减系数（0.15）
  c. 对二级相邻节点，增加分数 = s.分数 × 衰减系数²
步骤 3：合并原始分数和传播分数，生成新的排序
步骤 4：返回重打分后的结果列表
```

该方法的原理是：代码中核心函数具有较高的调用密度，被频繁调用的函数即使文本匹配度略低于噪声结果，也应当获得更高的排序优先级。

### 5.5 跨仓库 @ 宏路由语法

系统定义三种路由语法：

- `@repoName`：将搜索范围锁定到指定仓库（如 `@myProject`），同时从问题文本中移除该标记
- `@cross`：强制执行全量跨仓库搜索
- `#Q123`：引用历史沉淀条目

跨仓库搜索时，系统自动在结果文件路径前标注 `repoName:` 前缀，并在 LLM system prompt 中强制要求回答引用使用 `(repoName:path/file.ts:line)` 格式，确保跨仓库场景下引用来源清晰可辨。

### 5.6 意图×领域双维度回答模板矩阵

系统预定义 5 种回答模板（A/B/C/D/E），通过意图（6 维）和领域（6 类：通用/日志分析/堆栈分析/缺陷分析/编译构建/程序分析）的交叉乘积选择最匹配的模板：

```
模板 A：适用于排错场景（构建问题、日志分析）
   - 结构：错误定位 → 原因分析 → 解决方案 → 关键代码引用
模板 B：通用模板（默认）
   - 结构：功能说明 → 使用方式 → 关键代码 → 注意事项
模板 C：适用于存在质量缺陷的问题
   - 结构：缺陷定位 → 根因分析 → 修复建议 → 示例代码
模板 D：适用于架构/结构类问题
   - 结构：整体视图 → 模块划分 → 依赖关系 → 关键路径
模板 E：适用于影响分析
   - 结构：影响范围 → 调用链路 → 潜在风险 → 变更建议
```

选择逻辑：优先匹配 `intent_domain` 组合，其次匹配 `intent_general`，再次匹配 `domain`，最后使用模板 B。


### 5.7 全系统知识闭环数据流

上述各模块协同工作，形成完整的数据闭环：

```
                ┌─────────────────────────┐
                │   代码仓库（多仓库）       │
                │   repo-A  repo-B  repo-C │
                └──────────┬──────────────┘
                           │ AST 解析
                           ▼
                ┌─────────────────────────┐
                │   代码语义图谱构建         │
                │   （符号节点 + 调用关系边） │
                └──────────┬──────────────┘
                           │ 检索/上下文查询
                           ▼
    ┌─────────────────────────────────────────────┐
    │              问答交互层                       │
    │  用户问题 → 意图分类 → 差异化检索 →          │
    │  调用图重打分 → LLM 生成 → 结构化回答         │
    └─────────────────────┬───────────────────────┘
                          │ 自动入库
                          ▼
    ┌─────────────────────────────────────────────┐
    │            #Q 知识沉淀库                      │
    │   问答条目（#Q编号+双向关联）                  │
    │   校准答案（版本化+可追溯）                    │
    └─────────────────────┬───────────────────────┘
                          │ 已校准条目自动流入
                          ▼
    ┌─────────────────────────────────────────────┐
    │          Wiki 知识库                          │
    │   概览 · 模块文档 · 数据结构                  │
    │   常见踩坑（来自已校准问答）                   │
    └─────────────────────┬───────────────────────┘
                          │ 在后续问答中被引用
                          ▼
                   回到问答交互层（闭环）
```

上述闭环由三个子环构成：

- **外环（代码驱动）**：代码仓库变更 → 语义图谱更新 → 检索质量提升
- **内环（问答驱动）**：问答交互 → 校准沉淀 → 已校准答案直接复用
- **交叉环（知识流动）**：校准问答自动生成 Wiki → 用户浏览 Wiki 触发新问答 → 补充已有知识

三环结构使得系统的知识质量随使用时间持续提升，无需额外文档编写工作。

## 六、有益效果

与现有技术相比，本发明具有以下有益效果：

**1. 问答知识从"一次性交互"变为"可积累的知识资产"。** 每次问答自动分配 #Q 编号，支持双向关联和版本化校准，经校准的问答自动归档到 Wiki 知识库。同一问题被再次提问时直接返回已校准的标准答案，避免重复检索和生成。团队知识随着使用时间的增长持续积累，无需额外的文档编写工作。这是现有代码问答工具（Cody、Cursor、GitHub Copilot）所不具备的。

**2. 搜索范围自动判定，多仓库无感切换。** 通过自然语言理解自动判断搜索范围，无需用户手动选择仓库，跨仓库结果自动标注来源。

**3. 检索精度和排序质量得到增强。** 意图自适应检索和调用图传播重打分在不依赖核心机制独立实施的情况下，分别提升检索的精度和排序的质量，与核心机制配合使用时效果叠加。

### 独立保护点：#Q 问答知识结构化沉淀与复用

**核心内容：**

一种将代码问答交互中产生的知识进行结构化沉淀与复用的方法，包含以下四个协同环节构成的数据闭环：

（a）**自增编号**：每次问答交互自动分配一个全局递增的编号（#Q10000+），该编号作为问答条目的唯一标识，在后续交互中可通过 `#Q+编号` 语法引用；

（b）**双向关联**：在任意两个问答条目之间建立双向对称的关联链接，使得关联关系在两端互见，形成知识网络；

（c）**版本化校准**：支持对问答条目进行人工校准，每次校准生成包含版本号、校准人和校准理由的结构化记录，版本号自动递增，校准历史可追溯；

（d）**闭环复用与自动归档**：在后续问答交互中，系统自动解析输入文本中的 `#Q+编号` 引用标记，将被引用的历史条目（若已被校准则使用校准版本）的答案内容注入当前生成的上下文，同时在新条目与历史条目之间自动建立双向关联链接，并在当前问答完成后将结构和内容更新持久化；被标记为已校准的条目定期自动流入 Wiki 知识库，形成结构化知识沉淀页面。

**与现有技术的区别：**

现有代码问答工具（Sourcegraph Cody、Cursor AI、GitHub Copilot Chat）的每次问答均为独立会话，问答结果无法被编号、引用、关联和校准，属于"单向消耗型"交互。本发明通过四个环节的协同，使知识在"问答→沉淀→引用→新问答"的闭环中持续流动和增值。

**技术效果：**

- 已校准的问答在后续被引用时直接注入，避免重复检索和生成
- 引用关系自动建立双向链接，形成可追溯的知识演化网络
- 已校准问答自动流入 Wiki 知识库形成文档，无需额外编写
- 从"单向消耗"转变为"闭环积累"，随使用时间知识资产持续增长

### 从属增强特征 1：意图自适应检索

**核心内容：**

在上述独立保护点的基础上，进一步通过 LLM 分类器将用户问题实时分类到 6 种预定义代码意图（功能解释/定位/用法示例/排错/架构分析/影响分析），每种意图对应不同的检索词构造规则和结果扩展规则。当 LLM 不可用时自动降级到正则规则分类器。

**差异化策略示例：**

| 意图 | 检索词构造 | 结果扩展 |
|------|-----------|---------|
| 定位（where-is） | 仅精确符号名 | 最小化上下文 |
| 用法（how-to） | 符号名 + config/example/usage 提示词 | 展开调用者+被调用者 |
| 排错（why-error） | 先提取错误码，再提取符号名（两阶段） | 合并两阶段结果 |
| 影响（what-impact） | 符号名 + usage/reference | BFS 展开全部调用者（深度2） |

**与现有技术的区别：**
现有方案对所有类型问题采用统一检索策略。本特征是领域内首个面向代码问答的差异化检索方案。

**技术效果：**
不同类型问题获得适配的检索策略，避免统一策略在不同意图间的折中损失。

---

### 从属增强特征 2：调用图传播重打分

**核心内容：**

在上述独立保护点的基础上，进一步在混合检索（FTS5 + 向量 + RRF 融合）之后，沿代码调用关系图的边对 Top-N 结果进行分数传播——每个种子的分数按衰减系数 0.15 向直接调用者/被调用者传播，0.15² 向二级节点传播。

**与现有技术的区别：**
现有排序方法仅依赖文本匹配度或语义相似度。本特征引入代码特有的调用关系信号——被高频调用的核心函数即使在文本上匹配度略低，也能通过调用图传播获得更高的排序位置。这是文本检索和语义检索无法做到的。与通用的 PageRank 类算法不同，本特征针对代码检索场景设计了**阶段式耦合结构**（插入在 RRF 融合之后、LLM 上下文构建之前，限定传播种子数量为 Top-N），并通过**双向传播**（同时向调用者和被调用者传播）适配代码问答中"理解用法需展开下游被调用者、分析影响需追踪上游调用者"的差异化需求。

**技术效果：**
- 核心库函数排位提升，噪声结果排位下降
- 利用代码拓扑结构信号，无需额外训练
- 完全基于已有调用关系图，增量更新即可

---

### 从属增强特征 3：跨仓库动态范围限定

**核心内容：**

在上述独立保护点的基础上，进一步支持：通过 LLM 对问题进行搜索范围分类（单库/跨库对比/跨库调用链/全局搜索），当问题文本显式提到仓库名时自动限定范围，同时支持 `@repoName` 和 `@cross` 显式语法。

**与现有技术的区别：**
Sourcegraph Cody 的跨仓库搜索需企业级部署，Cursor/Aider 为单仓库设计。本特征在无需企业级基础设施的情况下，通过自然语言理解自动判定搜索范围。

**技术效果：**
多仓库场景下搜索范围自动判定，结果自动标注来源，无需用户手动选择。

## 八、具体实施方式

### 8.1 系统架构概述

本发明的实施系统按功能划分为四个层次，各层之间通过标准化的接口通信：

- **数据持久层**：负责代码语义图谱和问答知识的结构化存储，支持高效的图遍历查询和全文检索
- **索引与分析层**：负责从源代码中提取代码结构信息、构建语义图谱，并提供检索、上下文分析、影响分析等能力
- **问答与沉淀层**：负责意图分类、检索策略选择、调用图重打分、LLM 上下文构建和回答生成，以及 #Q 编号分配、关联链接和版本化校准
- **呈现层**：负责 Web 界面呈现、Wiki 知识库浏览、SSE 流式问答交互

系统支持多仓库部署——每个仓库独立构建语义图谱，问答时根据问题意图动态决定搜索范围。各层均支持水平扩展（多仓库并行索引、多实例负载均衡）。

以下为一种具体的组件选型和部署配置。

### 8.2 组件选型与部署配置

- **数据层**：使用 SQLite 数据库（Node.js 内置 `node:sqlite` 驱动），包含两套独立的数据库——codegraph 索引库（存储代码符号、调用关系图）和 qa 知识库（存储 #Q 条目、校准记录、关联链接）
- **服务层**：Express HTTP 服务器，提供 SSE 流式问答端点和 RESTful API
- **索引引擎**：基于 Tree-sitter 的 codegraph 引擎，提供多通道 FTS5 搜索和调用关系图查询
- **嵌入引擎**：通过 `@xenova/transformers` 加载本地 ONNX 模型（all-MiniLM-L6-v2，384 维），为代码符号和查询文本生成嵌入向量
- **LLM 推理**：兼容 OpenAI、DeepSeek、Anthropic 等主流 LLM API 接口
- **可选重排序引擎**：为提升最终排序质量，可配置一个独立的交叉编码器重排序服务，对 RRF 融合结果进行二次精排——该服务以子进程或独立容器方式部署，与主服务通过标准输入输出通信

### 8.4 关键参数

在本实施方式中：

- #Q 编号起始值：10000
- RRF 融合参数 k：60
- 调用图传播衰减系数：0.15
- 调用图传播种子数（Top-N）：3
- 向量嵌入维度：384（all-MiniLM-L6-v2）
- 向量搜索方式：暴力余弦相似度（内存加载）
- LLM 温度参数：0（意图分类）/ 0.1（回答生成）

### 8.5 应用场景

本系统适用于以下场景：

1. **团队级多仓库代码知识库**：团队在多个仓库上工作，通过问答交互逐步沉淀每个仓库的代码知识
2. **新成员入职培训**：新成员可通过问答快速了解仓库架构和关键模块
3. **代码评审辅助**：评审人员通过问答了解受影响的代码范围和调用链
4. **故障排查**：开发人员上传错误日志，系统自动分析定位问题根因
5. **知识交接**：通过 #Q 引用和双向关联链，团队成员可追溯问题的完整上下文

## 九、摘要

本发明公开了一种基于问答交互的代码知识结构化沉淀与复用方法及系统。该系统包括：自增编号模块，为每次问答交互分配全局唯一的 #Q 编号；双向关联模块，在问答条目之间建立双向对称的关联链接；版本化校准模块，支持人工校准并自动递增版本号；闭环复用模块，在后续问答中自动解析 `#Q+编号` 引用标记并将历史已校准答案注入生成上下文。在此基础上，还进一步包括意图自适应检索引擎和调用图传播重打分模块作为增强手段。本发明将代码问答与知识沉淀相结合，使团队问答知识从孤立会话转变为可引用、可校准、可复用的结构化知识资产。
