# 技术交底书

## 一、发明名称

一种人机协同的代码知识Wiki迭代闭环沉淀方法及系统

---

## 二、技术领域

本发明涉及软件开发工具技术领域，具体涉及一种基于大语言模型（LLM）的代码知识问答、跨仓库检索与问答知识结构化沉淀的方法及系统。

---

## 三、背景技术

### 3.1 现有技术概述

随着大语言模型（LLM）技术的快速发展，基于 LLM 的代码理解与问答工具（如 Sourcegraph Cody、Cursor AI、GitHub Copilot Chat 等）以及代码文档生成方案（如 DeepWiki、CodeWiki 等）已成为软件开发中的常用辅助手段。

然而，现有技术存在以下不足：

**（1）AI 生成的文档目录与团队实际业务模块脱节，导致 Wiki 使用率低下。** 以 DeepWiki 为代表的方案完全由 AI 自动将代码目录结构映射为文档目录，缺乏人工定义业务域层的能力——例如，一个涉及"订单支付"的业务流程可能分散在 checkout/、payment/、refund/ 等多个代码目录中，AI 无法理解这种跨目录的业务聚合。由此生成的 Wiki 目录仅反映代码物理结构而非团队心智模型，开发者难以快速找到所需信息，导致 Wiki 页面点击率极低，实际价值有限。

**（2）缺少人工主导与 AI 辅助的分层协作机制。** 现有方案的文档目录划分、内容生成完全由 AI 自主完成，人工无法定义顶层业务模块结构、无法配置代码目录与业务模块的映射关系，AI 也不具备将碎片化代码变更自动归集到正确业务模块的能力。人机权责不分层，导致生成的文档缺乏业务视角。

**（3）问答结果孤立，无法沉淀为团队知识。** 每次问答都是独立的会话，问答过程产生的知识无法被自动编号、引用、关联和校准。团队中不同成员可能反复提出相同问题，每次都需要重新执行检索和生成，无法复用已确认的正确答案。

**（4）检索策略单一，无法适应不同意图的代码问题。** 现有方案对"这个函数是做什么的"、"这个函数定义在哪里"、"这个函数怎么使用"、"为什么报这个错误"等不同类型的代码问题通常采用相同的检索策略（统一的 BM25 或向量检索），未针对不同意图优化检索词构造和结果扩展方式，检索精度受限。

**（5）未利用代码特有的调用关系优化检索排序。** 仅使用传统的文本匹配或语义匹配排序，忽略了代码中函数、模块之间的调用关系对相关性排序的提升作用——被大量调用的核心函数应当比同等文本匹配度的一般函数有更高的排序优先级。

**（6）团队知识资产绑定个人账号，人员流动导致知识流失。** 现有方案的问答记录、个性化配置均绑定用户个人账号，当团队成员离职后，其历史问答记录和校准贡献无法保留给团队；团队成员之间的问答经验相互不可见，无法形成积累效应。

#### 对比文献 1：中国专利 CN 120,723,297 A

**标题：** 一种基于大模型驱动代码反向生成设计书的方法及系统  
**申请人：** 中国实体（2025 年授权）

**核心内容：**
该专利提出了一种结合语法分析与大语言模型的代码反向生成设计书方案，通过构建知识图谱捕获显式依赖（AST 分析）和隐式依赖（LLM 提取），利用启发式搜索和向量聚类进行功能点挖掘，最终生成概要设计与详细设计文档。

**存在的缺陷：**
该专利依赖 LLM 提取"隐式依赖"来构建知识图谱，导致模块边界的确定存在不确定性和不可复现性，且其输出是固定的"设计书"格式文档，无法在后续问答中被引用、校准或沉淀为团队可复用的知识条目，缺少从"问答交互→知识沉淀"的闭环机制。

**与本发明的进一步对比：**
本发明的代码调用关系图构建（§5.5）与对比文献 1 的知识图谱构建存在本质区别——本发明完全基于**确定性语法分析**（Tree-sitter AST），不依赖任何 LLM 参与提取代码关系，同一份代码在任何环境下解析始终得到相同的图结构，具有完全可复现性。对比文献 1 依赖 LLM 提取"隐式依赖"，属于非确定性过程，同一份代码在不同 LLM 运行下可能得到不同的图谱结构。即使对比文献 1 也使用了 AST 分析，其叠加 LLM 隐式依赖提取的方式使得模块边界的确定引入了不可控的不确定性因素，而本发明的调用关系图构建全过程可验证、可复现，这是两者在代码分析方式上的根本差异。

#### 对比文献 2：美国专利 US 11,893,385 B2

**标题：** Methods and Systems for Automated Software Natural Language Documentation  
**申请人：** Open Weaver Inc.（现属 Google LLC，2021 年申请）

**核心内容：**
该专利提出了一种双 ML 模型的自动化自然语言文档生成方法，第一模型基于编程语言规范训练，将代码分解为 AST，第二模型基于开源代码训练，生成语句级文档、函数级文档和摘要输出。

**存在的缺陷：**
该专利采用自底向上的逐级文档组合方式（语句→函数→摘要），无法感知模块级的整体结构和跨模块的依赖关系，生成的文档缺乏架构层面的视图。且其为单向生成流程，生成的文档无法在后续问答中动态引用和校准，缺乏交互式知识沉淀的能力。

#### 对比文献 3：CodeWiki（学术论文，arXiv 2510.24428）

**标题：** CodeWiki: A Repository-level Multi-language Documentation Generation Framework  
**作者：** FPT Software AI Center（2025 年 11 月发布）

**核心内容：**
该论文提出了首个开源仓库级文档生成框架，采用层次化解构（hierarchical decomposition）+ 递归多智能体处理 + 多模态合成（文本 + 架构图）的方式，覆盖 7 种编程语言，从代码仓库自动生成结构化文档。

**存在的缺陷：**
该方法仍然是一次性的文档生成流程，未涉及"从问答交互中沉淀知识"的机制。多智能体递归处理的成本较高（每次生成均需多次 LLM 调用），且生成的文档与仓库绑定，不支持跨仓库搜索和引用。此外，该方法缺乏对问答结果的版本化校准和团队协作校准能力。

#### 对比文献 4：基于图结构的代码搜索重排序方法（学术论文，SIGIR 2022）

**标题：** CodeSearchNet Challenge: Graph-based Code Retrieval Re-ranking  
**核心内容：** 该研究提出在代码检索结果上应用图排序算法（基于函数调用图的 Personalized PageRank）进行重排序，利用代码的拓扑结构提升检索准确性。

**与本发明调用图传播重打分的区别：**
- 该方案使用全局的 Personalized PageRank，排序结果与查询无关，仅反映函数的全局重要性；而本发明采用**查询相关的局部传播**（仅从 RRF 融合后的 Top-3 种子出发传播），传播范围受查询结果限定
- 该方案的传播方向固定为调用者→被调用者（沿调用链正向），不支持双向传播；本发明支持**双向传播**（同时向调用者和被调用者），适配"理解用法需展开下游、分析影响需追踪上游"的差异化问答场景
- 该方案的传播强度在全图范围内均匀衰减，无参数限定；本发明通过**衰减系数 0.15 + 种子数 Top-N=3 的双参数约束**，将传播范围控制在局部，避免引入无关噪声

#### 对比文献 5：Sourcegraph 代码搜索分类系统（技术博客与公开文档，2024）

**标题：** Sourcegraph Code Search: Intent-aware Search Architecture  
**核心内容：** Sourcegraph 在代码搜索中引入了基于规则的意图分类系统，将查询分类为"符号搜索""全文搜索""文件路径搜索"等类型，并据此选择索引类型和排序策略。

**与本发明意图自适应检索的区别：**
- Sourcegraph 针对的是代码搜索引擎的通用搜索意图（符号/文本/路径），分类维度反映的是**索引结构**；本发明针对的是**问答交互场景**的 6 种差异化学**代码理解**意图（功能解释/定位/排错/用法/架构/影响），分类维度反映的是**开发者的认知目标**
- Sourcegraph 的策略差异主要体现为索引选择（哪个索引引擎响应查询）；本发明的策略差异体现在**检索词构造规则**和**结果扩展范围**两个维度，同一索引引擎内执行不同的检索策略
- 本发明包含 LLM 主分类 + 正则降级的**双阶段分类器**，Sourcegraph 为纯规则方案，无 LLM 分类层

综上所述，现有技术存在六个主要技术空白：

1. **人工无法定义业务模块层，AI 生成的文档目录与团队实际业务脱节**——导致 Wiki 使用率极低；
2. **缺少人工主导、AI 辅助的分层协同机制**——人机权责不分层，文档缺乏业务视角；
3. **缺少从问答交互到结构化知识沉淀的闭环机制**——问答知识无法自动编号、关联、校准和复用；
4. **缺少意图自适应的代码检索策略**——不同类型问题采用相同的检索方式，精度受限；
5. **缺少利用代码调用关系图优化搜索排序的手段**——文本匹配和语义匹配之外的代码特有的排序信号被忽略；
6. **团队知识资产绑定个人账号**——人员流动导致知识流失，团队经验无法积累。

本发明正是针对上述六个技术空白提出的完整解决方案。

## 四、发明内容

### 4.1 要解决的技术问题

**一、核心问题（由独立权利要求解决）：**

本发明要解决的根本技术问题是：**如何将团队在代码开发过程中产生的碎片化知识（包括人工业务经验、问答交互内容、代码变更信息等），以人工主导、AI 辅助的分层协同方式，结构化沉淀为团队可复用的 Wiki 知识资产，并形成持续迭代的闭环。**

在此基础上，核心权利要求进一步解决以下问题：
1. 如何构建人工定义业务模块、AI 仅负责底层执行的权责分层架构，解决 AI 全权生成目录与团队实际业务脱节的问题；
2. 如何将 AI 生成的增量知识通过待审缓冲池隔离，经人工审核后再回流至正式 Wiki，避免 AI 幻觉污染公共文档；
3. 如何将问答交互中产生的知识以自增编号、可关联、可校准的方式沉淀下来。

**二、增强问题（由从属权利要求解决）：**

4. 如何根据用户提问的意图自适应选择检索策略，提高代码检索的精度；
5. 如何利用代码调用关系图优化搜索结果排序，使核心函数获得更高的排位；
6. 如何在多仓库场景下自动判定搜索范围；
7. 如何确保校准后的知识在后续问答中直接复用（自增编号引用解析）；
8. 如何引导用户持续访问 Wiki 页面，形成问答→Wiki→新问答的正向循环（回答锚定 Wiki 引用）。

### 4.2 技术方案概述

本发明提出了一种以**人工主导、AI 辅助的分层协同机制**为架构核心、以**自增编号知识沉淀**为闭环驱动力、以**意图自适应检索**和**调用图传播重打分**为增强手段的代码知识 Wiki 闭环沉淀方法及系统。

**核心架构——人机分层协同机制：**
系统将知识管理划分为人工控制层和 AI 执行层，两者权责分离。人工层负责定义 Wiki 顶层业务模块结构、配置代码目录与业务模块的映射关系、审核 AI 产出的增量知识终审决策；AI 层仅读取人工预设的映射表，扫描绑定目录自动生成底层代码 Wiki 子文档，并将问答等交互中的增量知识归集到对应业务模块的待审区，无权修改业务层级结构或直接改写正式 Wiki。

**闭环驱动机制——自增编号知识沉淀子系统：**
每次问答交互自动分配全局唯一的自增编号，支持条目间的双向关联链接和版本化人工校准。问答条目按校准状态分流：**未校准条目**流入待审区等待审核；**已校准条目**（校准动作本身即包含人工审核，记录了校准人和校准理由）直通 Wiki 知识库，形成可浏览的结构化文档。

**缓冲保障机制——待审区隔离：**
AI 生成的增量知识（含未校准问答条目）先存入专属待审核待审区，经管理员完成去重、纠错、业务校验后方可更新正式 Wiki 及刷新向量检索索引，从根本上隔离 AI 原始输出与公共文档。

**增强机制一——意图自适应检索：**
通过 LLM 分类器将自然语言问题映射到 6 种预定义代码意图之一，每种意图对应不同的检索词构造策略和结果扩展策略。

**增强机制二——调用图传播重打分：**
在混合检索（FTS5 + 向量搜索 + RRF 融合）之后，沿代码调用关系图的边对 Top-N 结果进行分数传播和衰减重打分。

**增强机制三——跨仓库范围判定：**
支持 `@repoName` 指定仓库、`@cross` 跨仓库搜索，以及基于自然语言的自动范围分类。

其中，核心架构与闭环驱动机制共同构成完整的技术方案，各增强机制可分别或组合地与核心方案配合实施。

## 五、技术方案

### 5.0 系统模块总览

本系统按功能划分为四个核心模块，各模块协同工作形成完整的知识闭环：

```
┌─────────────────────────────────────────────────────────┐
│  ① 知识编排模块（人工控制层）                           │
│     ├─ 业务模块树定义（增删改业务域）                    │
│     └─ 代码目录映射配置（业务模块↔代码目录绑定）         │
├─────────────────────────────────────────────────────────┤
│  ② 代码索引模块                           ③ Wiki生成模块 │
│     ├─ AST解析（Tree-sitter，7语言）       ├─ 按映射表生成 │
│     ├─ 调用关系图构建                      ├─ 多目录聚合   │
│     ├─ FTS5 + 向量索引                     └─ 增量更新合并 │
│     └─ 跨文件符号消歧                                      │
├─────────────────────────────────────────────────────────┤
│  ④ QA模块（闭环沉淀核心）                                │
│     ├─ 意图分类 → 差异化检索 → 调用图重打分               │
│     ├─ 自增编号沉淀 + 待审区 + 审核终审                  │
│     └─ 锚定Wiki引用 → 引流回Wiki                          │
└─────────────────────────────────────────────────────────┘
         ↑ 知识编排指引AI执行，AI产出经待审区回流Wiki
```

其中，知识编排模块（本章 5.2）为人工控制层，定义了系统的业务骨架；代码索引模块（本章 5.5、5.6、5.7）提供代码级检索能力；Wiki生成模块（本章 5.2.4）按编排结果自动生成文档；QA模块（本章 5.1、5.3、5.4、5.8、5.9、5.10）完成问答交互、知识沉淀和闭环回流。四个模块的关系可概括为：**人工编排骨架 → AI索引代码 + 生成Wiki → 问答沉淀知识 → 审核后回流Wiki → 循环迭代**。

### 5.1 自增编号问答知识沉淀系统

#### 5.1.1 数据结构设计

系统维护两个核心数据库表：

**问答条目表（qa_entries）** 包含以下字段：自增编号 qid（全局递增）、会话标识 sessionId、关联仓库 repo、所属模块 module、原始问题 question、LLM 回答 answer、查询模式 mode（轻量/深度）、领域分类 domain、状态 status（活跃/归档/待审）、父条目 parent_qid、关联条目编号列表 related_qids、标签 tags、引用来源 sources、访问计数 visit_count、时间戳。

**校准答案表（calibrated_answers）** 包含以下字段：关联问答条目 ID qaEntryId、校准答案内容 answer、校准人标识 calibrator、校准理由 reason、版本号 version（首次为 1，每次校准递增）、时间戳。

#### 5.1.2 自增编号机制

系统在创建新问答条目时，从预设起始值（如 10000）开始递增，后续每次查询当前最大 qid 值并加 1 的方式生成新的编号。该编号全局唯一、单调递增，在后续问答中可通过约定的引用语法（如 `@123` 格式）引用，系统自动解析并注入被引用条目的上下文。

#### 5.1.3 双向关联链接机制

`linkEntries(sourceId, targetId)` 方法在两个问答条目之间建立双向对称关联：

```
步骤 1：获取源条目 source 和目标条目 target
步骤 2：将 targetId 添加到 source.relatedIds（如已存在则跳过）
步骤 3：将 sourceId 添加到 target.relatedIds（如已存在则跳过）
步骤 4：同时更新两条记录的时间戳
```

该机制确保关联关系始终是对称的，任意一端建立的关联在另一端自动可见。

#### 5.1.4 版本化校准机制

`upsertCalibratedAnswer()` 方法支持对任意问答条目进行人工校准：

```
首次校准：创建校准记录，version = 1，记录校准人和校准理由
再次校准：version 自动 +1，保留首次校准的 createdAt，更新 updatedAt
```

每次校准都会递增版本号，使得校准历史可追溯。校准操作不仅修正问答内容的技术正确性，还同时校验条目的业务模块归类匹配性，确保校准后的条目在归类上准确无误后方可通过。

#### 5.1.5 引用自动解析

在后续问答输入中，系统自动通过正则表达式解析引用标记，获取被引用的条目信息，自动建立双向关联链接，并将被引用条目的上下文注入 LLM 提示词。

### 5.2 人机分层协同架构

#### 5.2.1 层次划分

系统将代码知识管理划分为两个权责分明的层次：

**（a）人工控制层**——由仓库管理员或团队技术负责人操作，拥有顶层决策权限：
- 定义 Wiki 业务模块结构：创建、修改、合并、删除顶层业务域模块，搭建 Wiki 整体阅读骨架
- 配置代码映射：为每个业务模块指定关联的代码仓库、目录路径及文件过滤规则
- 审核决策：对 AI 产出的增量知识进行去重、纠错、业务校验终审
- 补充人工知识：补充跨代码包的架构方案、业务约束规则、线上踩坑经验等 AI 无法自主识别的知识

**（b）AI 执行层**——系统自动运行，仅拥有预设范围内的执行权限：
- 按映射表自动生成文档：读取人工预设的业务模块-代码目录映射表，扫描绑定目录中的代码文件，为每个模块自动生成底层代码 Wiki 子文档
- 自动归集增量：将问答内容、代码变更提交、报错记录等信息依据映射关系自动归集至对应的业务模块
- 萃取待审条目：将对话交互中产生的增量知识提取后送入待审区，无权直接修改正式 Wiki

#### 5.2.2 分层交互流程

```
人工：定义业务模块 A/B/C → 配置各模块代码目录映射 → 确认骨架
                                ↓
AI  ：读取映射表 → 扫描绑定目录 → 生成模块文档（填充内容）
                                ↓
用户：提问/提交代码/Wiki 批注 → 产生增量知识
                                ↓
AI  ：萃取增量 → 按映射归集 → 送入对应模块的待审区
                                ↓
人工：审核待审条目 → 通过/驳回/修改 → 终审确认
                                ↓
系统：审核通过 → 更新 Wiki 模块 → 刷新向量索引
```

#### 5.2.3 分层权限控制

系统实现三角色权限模型：

| 角色 | 操作权限 |
|------|---------|
| 只读成员 | 浏览 Wiki、发起问答、查看公共问答池 |
| 编辑成员 | 上述权限 + Wiki 内容编辑、待审条目审核、补充人工知识 |
| 仓库管理员 | 上述权限 + 业务模块定义/修改/删除、代码目录映射配置、角色管理 |

所有顶层结构性变更（新增模块、调整映射、修改业务层级）仅限仓库管理员操作，AI 在任何角色下均不拥有这些权限。

#### 5.2.4 AI 文档生成机制

AI 根据人工预设的业务模块-代码目录映射表，自动生成模块底层 Wiki 文档，具体实现方式如下：

**（a）Prompt 模板化生成**：系统为每类代码结构（函数定义、类定义、模块导出、配置文件、测试用例）预定义文档模板；AI 读取绑定目录中的代码文件后，按文件类型匹配对应模板，将代码符号信息和上下文填入模板生成结构化文档段落。例如，一个函数会被生成包含"功能说明→参数说明→返回值→调用示例→注意事项"的标准化段落。

**（b）多目录聚合**：当人工将一个业务模块映射到多个代码目录时（如"订单支付"模块映射到 checkout/、payment/、refund/ 三个目录），AI 分别扫描每个绑定目录生成子文档片段，再按人工指定的业务逻辑顺序（如"创建订单→支付→退款"流程）合并为完整模块文档。

**（c）章节结构策略**：AI 生成文档的章节结构采用半模板化策略——顶层章节名称继承人工定义的业务模块名；二级章节按代码子目录或功能域自动生成；三级及以下细节内容由 AI 根据代码结构灵活组织。人工可通过修改映射配置中的"章节顺序"参数调整生成结构。

**（d）增量更新合并**：当绑定目录中的代码发生变更时，AI 仅重新生成受影响子文档片段，通过与已有 Wiki 内容的逐段 diff 对比，仅替换发生变化的段落，保留人工补充的同模块非代码知识（如架构说明、踩坑记录）。

#### 5.3.1 待审区设计

系统为每个仓库的每个业务模块维护一个独立的待审核待审区，支持以下操作：

- **写入待审**：AI 从问答对话、Wiki 批注、代码提交注释等渠道萃取的增量知识，自动标记来源渠道和置信度后写入对应模块的待审区
- **待审状态流转**：新建（draft）→ 审核中（reviewing）→ 已通过（approved）/ 已驳回（rejected）
- **版本对比**：待审内容与正式 Wiki 当前版本的差异视图，辅助审核人员快速判断变更内容

#### 5.3.2 审核流程

```
步骤 1：系统通知管理员对应模块有新增待审条目
步骤 2：管理员查看待审内容及与正式 Wiki 的差异对比
步骤 3：管理员执行以下三种操作之一：
  a. 通过——待审内容合并至正式 Wiki 对应模块
  b. 修改——编辑待审内容后再通过
  c. 驳回——退回待审条目并附驳回理由（AI 可据此修正后重新提交）
步骤 4：审核通过后，系统自动触发：
  a. 更新对应 Wiki 模块的正式内容
  b. 刷新向量检索索引（使更新内容立即可检索）
  c. 将审核记录写入操作日志（追踪知识变更历史）
```

#### 5.3.3 缓冲隔离效果

该机制确保：
- AI 原始输出与正式 Wiki 文档之间始终存在审核隔离层
- 待审内容未获人工确认前，不会出现在公共问答的检索结果中
- 即使 LLM 产生幻觉或错误信息，也不会直接污染团队公共知识资产

### 5.4 6 维意图自适应代码检索方法

#### 5.4.1 意图定义

系统预定义 6 种面向代码问答的意图：

| 意图标识 | 含义 | 典型问题示例 |
|---------|------|-------------|
| what-is | 功能解释 | "这个函数是做什么的"、"解释一下这段代码" |
| where-is | 定位 | "这个类定义在哪里"、"实现在哪" |
| how-to | 用法示例 | "怎么用这个 API"、"如何调用" |
| why-error | 排错分析 | "为什么报这个错"、"分析这个堆栈" |
| what-structure | 架构分析 | "整体架构是什么"、"模块关系图" |
| what-impact | 影响分析 | "改了 A 会影响谁"、"调用链分析" |

#### 5.4.2 两阶段意图分类器

系统采用主备两阶段分类器：

**第一阶段（主分类器）：LLM 分类**

通过向 LLM 发送带 JSON Schema 的 system prompt，要求返回结构化的分类结果：

```
System Prompt: "你是一个代码意图分析师。根据问题和可用仓库，返回 JSON 格式：
  意图（intent）：what-is | where-is | how-to | why-error | what-structure | what-impact
  范围（scope）：single | cross-compare | cross-call | global-search
  返回格式：{"intent":"what-is","scope":"single","reasoning":"..."}"
```

设置 temperature=0 保证分类的确定性。response_format 设置为 json_object 确保输出可解析。

**第二阶段（降级分类器）：规则匹配 + 置信度打分**

当 LLM 分类不可用（网络异常、API 不可达等）时，使用预定义的正则规则集合进行意图分类。每条规则包含正则模式和优先级，最终选择匹配规则中优先级最高的意图作为输出。同时计算置信度分数：

```
置信度 = 匹配比例 × 0.5 + 优先级因子 × 0.5
其中优先级因子 = min(1, (最高优先级 + 1) / 6)
```

置信度低于 0.3 时默认使用 what-is 意图。

#### 5.4.3 差异化检索策略

根据分类得到的意图，系统从预定义的检索策略表中选择对应的策略执行。每种策略包含检索词构造规则和结果扩展规则两部分：

**（a）what-is（功能解释）策略：**
- 检索词构造：使用提取的符号名 + 上下文关键词（当符号名为空时使用去除代码关键词后的自然语言词）
- 结果扩展：对 Top-N 结果调用 codegraph_context 工具展开完整定义内容

**（b）where-is（定位）策略：**
- 检索词构造：仅使用精确符号名，不添加任何扩展词
- 结果扩展：最小化扩展路径，仅返回定位所需的最小上下文

**（c）how-to（用法示例）策略：**
- 检索词构造：使用符号名 + 预定义提示词组合（config、example、usage、setup、init）
- 结果扩展：对 Top-N 结果同时展开调用者和被调用者信息，展示调用链上下文

**（d）why-error（排错）策略：**
- 检索词构造：两阶段构造——第一阶段提取大写错误码（如 EACCES、TIMEOUT）和错误值模式，第二阶段提取常规符号名
- 搜索执行：第一阶段对错误码进行文本 grep，第二阶段对符号名进行 codegraph 搜索
- 结果扩展：合并两阶段结果后展开上下文

**（e）what-structure（架构）策略：**
- 检索词构造：使用符号名 + 类型提示词（module、interface、type、enum）+ 通用结构词（index、main、README、package）
- 结果扩展：展开结构定义和类型层次

**（f）what-impact（影响分析）策略：**
- 检索词构造：使用符号名 + usage、reference 等影响相关词
- 结果扩展：对 Top-1 结果展开其全部调用者和影响范围（BFS，深度 2）

#### 5.4.4 搜索范围分类

系统同时分类搜索范围，决定在哪些仓库中执行检索：

```
规则 1：问题中出现多个仓库名 → cross-compare（跨库对比）
规则 2：问题中出现单个仓库名 → single（单库）
规则 3：影响/调用链关键词 → impact / cross-call
规则 4：对比类关键词 → cross-compare
规则 5：跨库引用关键词 → cross-call
规则 6：模糊搜索关键词 → global-search
规则 7：以上均不匹配 → single（默认）
```

### 5.5 代码调用关系图的构建

本发明所依赖的代码调用关系图通过以下方式构建：

**（a）结构解析层**：基于 Tree-sitter 对各语言源代码进行 AST 解析，提取函数定义与调用、类定义与继承、模块导入/导出、类型定义、被赋值为函数的变量等结构单元。目前支持 JavaScript/TypeScript、Python、Go、Rust、Java、C/C++、Ruby 共 7 种编程语言，每种语言对应独立的 Tree-sitter 语法解析器和查询规则。

**（b）跨文件符号解析消歧**：将分散在不同文件中的同名符号关联为同一节点，采用三级消歧策略：
- **优先匹配**：使用全限定名（如 `PackageName.Module.Function`）进行匹配
- **次优先**：对仅知短名的符号，结合当前文件的 import/require/using 语句推断完整路径
- **兜底**：同一文件中出现同名符号时，按作用域链（块级→函数级→模块级）确定最近绑定
- **动态调用处理**：对动态语言中的间接调用（如 `obj[method]()`、反射调用）仅记录调用站点，标记为"不完全解析"，不强制关联目标节点

**（c）图结构**：覆盖全仓库的统一有向图，节点包含符号名、全限定名、文件路径、行号范围、类型标签（function/class/interface/module/variable）、入度（被调用次数）、出度（调用其他符号次数）。以邻接表形式持久化存储。

**（d）增量更新**：采用以下机制确保调用关系图随代码变更高效更新：
- **变更检测**：通过 git post-commit hook 捕获提交变更，传递变更文件列表（含文件路径、变更类型（新增/修改/删除））至更新模块；非 git 环境通过文件系统监听（inotify/FSEvents）驱动。
- **依赖边级联策略**：当节点 N 被标记为"修改"时，其所有入射边（其他节点→N）和出射边（N→其他节点）一并失效。失效后对 N 所在文件重新执行 AST 解析，重建其出射边（调用了谁）；入射边（谁调用了 N）由调用者所在文件的解析结果重建——若调用者所在文件本次未变更，则复用历史解析缓存中该文件的调用信息。
- **节点映射算法**：以文件路径 + 符号全限定名作为节点唯一标识。变更文件重新解析后，对每个新解析出的符号，若其标识已存在于旧图则视为"修改"（更新已有节点属性），若不存在则视为"新增"（添加新节点），若旧图节点在当前文件解析结果中未出现则视为"删除"（递归删除该节点及其关联边）。
- **变更触发**：由 git post-commit hook 或文件系统监听事件驱动，无需全量重建。

### 5.6 调用图传播重打分方法

本发明的混合搜索管道采用以下阶段式架构：

```
阶段 1：FTS5 全文搜索（codegraph_search）
阶段 2：向量语义搜索（cosine 相似度）
阶段 3：RRF 融合（融合阶段 1 和阶段 2，k=60）
阶段 4：调用图传播重打分
阶段 5（可选）：交叉编码器重排序（Python 子进程）
```

其中阶段 4 的调用图传播重打分方法如下：

```
输入：RRF 融合后的排序结果列表 R
参数：Top-N = 3（传播种子数），衰减系数 = 0.15

步骤 1：取 R 的前 N 个结果作为传播种子 S，初始化已访问节点集合 V = S
步骤 2：对每个种子 s ∈ S：
  a. 查询代码调用关系图，获取 s 的直接调用者和被调用者
  b. 对每个相邻节点 n ∉ V，增加分数 = s.分数 × 衰减系数（0.15），将 n 加入 V
  c. 对每个二级相邻节点 m ∉ V（相邻节点的相邻节点），增加分数 = s.分数 × 衰减系数²，将 m 加入 V
  （注：已访问节点集合 V 用于避免循环依赖导致重复传播，无论通过何种传播路径，同一节点至多被处理一次）
步骤 3：合并原始分数和传播分数，生成新的排序
步骤 4：返回重打分后的结果列表
```

该方法的原理是：代码中核心函数具有较高的调用密度，被频繁调用的函数即使文本匹配度略低于噪声结果，也应当获得更高的排序优先级。

### 5.7 跨仓库 @ 宏路由语法

系统定义三种路由语法：

- `@repoName`：将搜索范围锁定到指定仓库（如 `@myProject`），同时从问题文本中移除该标记
- `@cross`：强制执行全量跨仓库搜索
- `@编号`：引用历史沉淀条目（如 `@12345`）

跨仓库搜索时，系统自动在结果文件路径前标注 `repoName:` 前缀，并在 LLM system prompt 中强制要求回答引用使用 `(repoName:path/file.ts:line)` 格式，确保跨仓库场景下引用来源清晰可辨。

### 5.8 意图驱动的回答模板选择

系统预定义 5 种回答模板（A/B/C/D/E），根据意图分类结果直接选择匹配模板：

| 意图 | 模板 | 答案结构 |
|------|------|---------|
| what-is（功能解释） | 模板 B | 功能说明 → 使用方式 → 关键代码 → 注意事项 |
| where-is（定位） | 模板 B（精简） | 位置说明 → 文件路径 → 行号 |
| how-to（用法示例） | 模板 B（展开） | 功能说明 → 使用方式 → 调用示例 → 注意事项 |
| why-error（排错） | 模板 A | 错误定位 → 原因分析 → 解决方案 → 关键代码引用 |
| what-structure（架构） | 模板 D | 整体视图 → 模块划分 → 依赖关系 → 关键路径 |
| what-impact（影响分析） | 模板 E | 影响范围 → 调用链路 → 潜在风险 → 变更建议 |

选择逻辑：依据意图分类结果直接映射到对应模板；无法识别时默认使用模板 B。


### 5.9 仓库级公共问答池

#### 5.9.1 资产归属模型

与现有工具（DeepWiki、Cursor AI）将问答记录绑定个人账号不同，本发明中每条问答条目永久归属于其所关联的**代码仓库**和**业务模块**，而非提问者个人。即使提问者离职或注销账号，其贡献的问答条目及其校准记录仍然保留在团队知识库中，不会丢失。

#### 5.9.2 全员可见的公共访问

- 所有归属同一仓库的合规问答条目（包括历史校准版本）在对应 Wiki 业务模块页面中常驻展示，成为团队全员可见的公共资产
- 新成员可直接检索和浏览历史问答记录，复用前人踩坑经验，无需重复提问
- 问答条目的访问计数和引用计数（被 `@编号` 引用次数）可作为知识价值度量的参考指标

### 5.10 回答强制锚定 Wiki 引用

为持续引导用户访问 Wiki 页面，形成"问答→Wiki→新问答"的正向循环，系统在 LLM 回答生成阶段实施以下机制：

- **回答内置 Wiki 章节链接**：LLM 回答中引用代码文件时，自动转换为对应 Wiki 模块页面的锚点链接（如 `Wiki: 订单支付模块 → 退款流程`），用户点击即可跳转至 Wiki 对应章节
- **来源显式标注**：回答底部自动列出本次回答引用的 Wiki 模块列表，附带模块完整度标识（有序/待完善/缺失），激励用户补充完善
- **跨会话溯源**：当回答引用历史已校准条目时，同时标注该条目的原始提问时间和校准人，增强可信度

### 5.11 全系统知识闭环数据流

上述各模块协同工作，形成完整的数据闭环：

```
                ┌─────────────────────────────┐
                │   人工控制层（顶层决策）       │
                │   ├─ 定义业务模块结构         │
                │   ├─ 配置代码目录映射         │
                │   └─ 审核终审                 │
                └──────────┬──────────────────┘
                           │ 人工预设映射表
                           ▼
                ┌─────────────────────────────┐
                │   代码仓库（多仓库）           │
                │   repo-A  repo-B  repo-C     │
                └──────────┬──────────────────┘
                           │ AST 解析
                           ▼
                ┌─────────────────────────────┐
                │   代码调用关系图构建             │
                │   （代码符号节点 + 调用关系边） │
                │   （独立于 Wiki 模块，仅供检索）│
                └──────────┬──────────────────┘
                           │ 检索/上下文查询
                           ▼
    ┌─────────────────────────────────────────────────┐
    │              问答交互层                          │
    │  用户问题 → 意图分类 → 差异化检索 →              │
    │  调用图重打分 → LLM 生成 → 锚定 Wiki 引用        │
    └──────────────────────┬──────────────────────────┘
                           │ 自动入库
                           ▼
    ┌─────────────────────────────────────────────────┐
    │   自增编号知识沉淀库                              │
    │   问答条目（自增编号 + 双向关联）                  │
    │   校准答案（版本化 + 可追溯）                     │
    └──────────┬───────────┬───────────────────────────┘
               │ 未校准条目  │ 已校准条目
               ▼             ▼
    ┌─────────────────────────────────────────────────┐
    │   待审区（仅接收未校准条目）                  │
    │   ├─ 待审核待审区（draft）                       │
    │   ├─ 审核中（reviewing）                         │
    │   └─ 审核通过后更新 Wiki                         │
    └──────────────────────┬──────────────────────────┘
                           │ 人工审核通过
                           ▼

    ┌─────────────────────────────────────────────────┐
    │   已校准条目直通 Wiki（跳过待审区）               │
    │   （校准动作本身即为一次人工审核）                │
    └──────────────────────┬──────────────────────────┘
                           │ 校准即通过
                           ▼
    ┌─────────────────────────────────────────────────┐
    │   Wiki 知识库（人工业务模块骨架 + AI 填充内容）    │
    │   概览 · 业务模块文档 · 数据结构                  │
    │   常见踩坑（来自已校准问答）· 公共问答池           │
    └──────────────────────┬──────────────────────────┘
                           │ 在后续问答中被引用（锚定 Wiki 链接）
                           ▼
                   回到问答交互层（闭环）
```

上述闭环由四个子环构成：

- **业务层环（人工驱动）**：管理员定义/调整业务模块和映射 → AI 按新映射重新生成文档 → 持续匹配团队实际业务
- **代码层环（代码驱动）**：代码仓库变更 → 语义图谱更新 → 检索质量提升
- **问答层环（交互驱动）**：问答交互 → 待审缓冲 → 人工审核 → 已校准答案复用
- **知识流动环（交叉驱动）**：审核通过的问答流入 Wiki → 用户浏览 Wiki 触发新问答 → 补充已有知识

四环结构使得系统的知识质量随使用时间持续提升，且人工始终掌握顶层结构的主导权。

在上述数据流中，**自增编号问答条目库被定义为主数据源（Source of Truth）**，Wiki 知识库为其只读衍生展示视图。对 Wiki 内容的任何编辑均需提交变更申请，经审核后回写入问答条目库再同步至 Wiki，确保两套载体始终一致，避免双向覆盖冲突。

## 六、有益效果

与现有技术相比，本发明具有以下有益效果：

**1. 文档结构贴合团队实际业务，解决 Wiki 低点击率问题。** 通过人工定义业务模块+AI 填充底层内容的分层协同机制，Wiki 目录结构反映的是团队心智模型而非代码物理结构。开发者能够按业务直觉快速定位到所需文档，从根本上解决 DeepWiki 等方案中 AI 生成目录与业务脱节导致的 Wiki 低使用率问题。

**2. 问答知识从"一次性交互"变为"可积累的知识资产"。** 每次问答自动分配自增编号，支持双向关联和版本化校准，经校准的问答通过待审缓冲审核后归档到 Wiki 知识库。同一问题被再次提问时直接返回已校准的标准答案，避免重复检索和生成。团队知识随着使用时间的增长持续积累。

**3. 知识资产归属仓库而非个人，人员流动不丢失。** 所有问答、校准记录、Wiki 内容均绑定代码仓库，不因成员离职而丢失。团队成员之间历史问答全员可见，形成积累效应。

**4. 搜索范围自动判定，多仓库无感切换。** 通过自然语言理解自动判断搜索范围，无需用户手动选择仓库，跨仓库结果自动标注来源。

**5. 检索精度和排序质量得到增强。** 意图自适应检索和调用图传播重打分在不依赖核心机制独立实施的情况下，分别提升检索的精度和排序的质量，与核心机制配合使用时效果叠加。

### 独立保护点 1：人机分层协同的代码知识 Wiki 迭代闭环沉淀方法

**核心内容：**

一种代码知识 Wiki 迭代闭环沉淀方法，包含以下七个协同环节构成的数据闭环：

（a）**人工配置业务模块结构**：仓库管理员定义 Wiki 顶层业务模块，配置每个业务模块与代码仓库中特定目录/文件的映射关系，搭建 Wiki 整体阅读骨架；

（b）**AI 按映射生成底层文档**：AI 读取人工预设的业务模块-代码目录映射表，扫描绑定目录中的代码文件，按模块自动生成底层代码 Wiki 子文档，填充模块内容；

（c）**问答交互增量采集**：通过用户问答交互渠道，提取新增技术增量知识；

（d）**待审缓冲隔离**：增量知识存入专属待审核待审区，隔离 AI 原始输出与正式 Wiki 文档，未审核内容不进入公共检索；

（e）**人工审核终审**：管理员对待审条目完成去重、纠错、业务校验，执行通过/修改/驳回三种操作之一；

（f）**回流更新**：审核通过后同步更新对应 Wiki 模块的正式内容，刷新向量检索索引；

（g）**闭环复用**：更新后的 Wiki 文档作为全体团队查询、提问的统一权威基准，后续问答交互中 LLM 回答强制锚定 Wiki 原文章节链接，引导用户持续访问 Wiki，开启下一轮迭代。

**与现有技术的区别：**

现有方案（DeepWiki、CodeWiki 等）均由 AI 全权主导文档目录划分和内容生成，人工无法干预顶层结构，导致生成的文档目录仅反映代码物理结构而非团队实际业务模块，Wiki 页面点击率极低。本发明首次提出人工与 AI 权责分层的协同架构——人工掌握业务模块定义、代码映射配置和内容终审的顶层决策权，AI 仅负责按映射表执行底层文档生成和知识归集，无权修改业务层级。同时，现有方案均为单向生成流程（代码→文档），本发明实现完整的"问答→待审→审核→Wiki→新问答"迭代闭环。

**技术效果：**

- Wiki 目录反映团队业务心智模型而非代码物理结构，文档可用性和点击率大幅提升
- AI 幻觉不会直接污染公共文档（待审缓冲隔离）
- 知识在闭环中持续增值，人工始终掌握顶层主导权
- 问答交互成为知识采集手段，非系统核心——可替换为批注、提交注释等其他采集方式，扩大保护范围

### 独立保护点 2：人机协同知识 Wiki 迭代闭环沉淀系统

**核心内容：**

一种实施权利要求 1 所述方法的人机协同知识 Wiki 迭代闭环沉淀系统，包含与方法各步骤相对应的系统模块：

（a）**人工配置管理模块**，对应于权利要求 1 步骤（a），提供业务模块结构定义、代码目录映射配置和审核操作的可视化界面，仅仓库管理员拥有业务层级修改权限；

（b）**AI 文档生成模块**，对应于权利要求 1 步骤（b），读取人工预设的映射表，扫描绑定目录中的代码文件并按模块自动生成底层 Wiki 子文档；

（c）**问答交互模块**，对应于权利要求 1 步骤（c），接收用户自然语言问题，执行意图分类、差异化检索、调用图重打分和 LLM 回答生成，并将回答自动关联至对应业务模块；

（d）**待审管理模块**，对应于权利要求 1 步骤（d）（e），管理各模块的待审核待审区（状态流转、版本差异对比、审核操作、老化策略），审核通过后自动触发回流更新；

（e）**回流更新模块**，对应于权利要求 1 步骤（f）（g），同步更新 Wiki 正式内容和向量检索索引，并在后续问答中强制锚定 Wiki 引用。

现有方案（DeepWiki、Cursor AI）缺乏系统的架构支持权利要求 1 所述的人机分层协同闭环方法。本发明的系统模块与权利要求 1 的方法步骤一一对应，形成完整的"人工配置→AI 生成→问答采集→审核→回流复用"的技术实现载体。

**技术效果：**

- 系统模块与方法步骤的严格对应关系确保技术方案的可实施性
- 各模块独立部署和扩展，支持不同团队灵活配置

### 从属增强特征 1：自增编号问答知识双向关联与版本化校准

**核心内容：**

在上述独立保护点的基础上，进一步通过以下机制增强知识沉淀的可追溯性和可复用性：

（a）**自增编号**：每次问答交互自动分配全局唯一的自增编号，作为知识条目的持久标识，在后续交互中可通过约定的引用语法引用；

（b）**双向关联**：在两个知识条目之间建立双向对称关联链接，任意一端建立的关联在另一端自动可见，形成可追溯的知识演化网络；

（c）**版本化校准**：支持对知识条目进行人工校准，每次校准生成包含版本号、校准人和校准理由的结构化记录，版本号自动递增，校准历史可追溯；

（d）**自动引用解析**：在后续问答中自动解析引用标记，获取被引用条目的上下文注入 LLM 提示词，实现已校准知识的直接复用。引用解析器在注入前强制校验被引用条目的状态——仅当条目同时满足以下条件时方允许注入：
  - 仓库归属校验：条目所属仓库在当前搜索范围内（或显式声明跨仓库引用）
  - 审核状态校验：条目状态为"已校准"或"已审核通过"
  - 生命周期校验：条目状态为"活跃"而非"已归档"或"已过期"
  任一校验不通过时，系统返回提示信息而非注入条目内容，确保 LLM 使用的始终是最新、合规、有效的知识。

**与现有技术的区别：**

现有工具的问答均为独立会话，无法编号、引用和校准。本特征首次在代码问答领域实现全局自增编号 + 双向关联 + 版本化校准的组合机制，使知识沉淀从"一次性记录"升级为"结构化知识网络"。

**技术效果：**

- 已校准知识在后续问答中直接注入复用，避免重复检索
- 引用关系自动双向关联，知识演化路径可追溯
- 与待审缓冲机制协同：未校准条目不进入公共检索，已校准条目自动流向 Wiki

### 从属增强特征 2：待审区置信度分级与智能通知机制

**核心内容：**

在上述独立保护点的基础上，进一步包含以下具体实现参数和机制：

（a）**置信度分级**：AI 写入待审区时依据来源渠道和内容匹配度自动计算置信度分数（0-1），分为高置信度（≥0.8，直接进入审核队列）、中置信度（0.5-0.8，标记待审）、低置信度（<0.5，标记需人工重点核查）三个等级，不同等级影响审核排序优先级；

（b）**待审老化策略**：待审条目超过预设天数（默认 30 天）未被处理时，系统自动发送催办通知至管理员；超过 90 天仍未处理的低置信度待审条目自动归档为"已过期"，不再占用审核队列；

（c）**审核通知触发条件**：系统仅在以下条件之一满足时推送审核通知——（i）待审区新增高置信度条目、（ii）待审区累计未审核条目超过预设阈值（默认 20 条）、（iii）同一业务模块在 7 天内无审核记录；

（d）**版本差异对比**：审核界面提供待审内容与正式 Wiki 当前版本的差异对比视图，采用逐行差异高亮（类似 unified diff），辅助审核人员快速判断变更范围；

（e）**并发校准锁**：当同一待审条目被多人同时操作时，系统采用乐观锁机制（基于版本号字段），后提交者的操作被拒绝并提示刷新后重试。

**与现有技术的区别：**

现有方案（DeepWiki、CodeWiki）中 AI 生成内容直接发布为正式文档，无隔离审核环节。本特征不仅在 AI 产出与正式文档之间设置强制审核层，更进一步通过置信度分级、老化策略和智能通知机制解决了"AI 产出量大但审核人力有限"的实际工程问题。

**技术效果：**

- AI 输出与公共文档之间的强制审核隔离
- 置信度分级引导审核人力优先处理高质量待审条目
- 老化策略防止待审区无限积压
- 乐观锁确保并发审核的数据一致性

### 从属增强特征 3：仓库级全员公共问答池

**核心内容：**

在上述独立保护点的基础上，进一步包含以下公共知识共享机制：

（a）**仓库归属**：所有问答条目永久绑定至其关联的代码仓库和业务模块，不因提问者个人状态变化而丢失；

（b）**页面集成**：在 Wiki 业务模块页面中常驻展示归属于该模块的全部历史问答列表；

（c）**可度量**：问答条目的访问计数和引用计数作为知识价值的参考指标，高价值问答自然浮现在前列。

**与现有技术的区别：**

现有方案问答记录仅限提问者个人查看，团队其他成员无法检索和复用。本特征将问答记录提升为仓库级公共资产，实现团队知识的全员可见和持续复用。

**技术效果：**

- 新成员直接检索历史问答，复用前人踩坑经验
- 知识价值可度量（访问/引用计数）
- 团队知识积累不依赖个人主动性

### 从属增强特征 4：回答强制锚定 Wiki 引用

**核心内容：**

在上述独立保护点的基础上，LLM 回答生成阶段进一步包含以下锚定机制：

（a）**Wiki 章节链接**：LLM 回答中引用代码或概念时，自动转换为对应 Wiki 模块页面的章节锚点链接；

（b）**来源显式标注**：回答底部自动列出本次引用的 Wiki 模块列表及各模块完整度标识（有序/待完善/缺失）；

（c）**跨会话溯源**：当回答引用历史已校准条目时，同步标注原始提问时间和校准人。

**与现有技术的区别：**

现有方案（Cursor、Cody）的回答引用直接指向代码文件和行号，不会关联 Wiki 文档页面。本特征通过强制锚定 Wiki 引用，使每次问答都成为引导用户访问 Wiki 的入口，形成"问答→Wiki→新问答"的正向循环，持续提升 Wiki 的活跃度和覆盖率。

**技术效果：**

- 每次问答交互均为 Wiki 引流，解决 Wiki 无人浏览的行业痛点
- 用户可一键从回答跳转至 Wiki 对应章节获取完整上下文
- Wiki 模块完整度可视化，激励团队主动补充

### 从属增强特征 5：意图自适应代码检索

**核心内容：**

在上述独立保护点的基础上，进一步通过 LLM 分类器将用户问题实时分类到 6 种预定义代码意图（功能解释/定位/用法示例/排错/架构分析/影响分析），每种意图对应不同的检索词构造规则和结果扩展规则。当 LLM 不可用时自动降级到正则规则分类器。

**差异化策略示例：**

| 意图 | 检索词构造 | 结果扩展 |
|------|-----------|---------|
| 定位（where-is） | 仅精确符号名 | 最小化上下文 |
| 用法（how-to） | 符号名 + config/example/usage 提示词 | 展开调用者+被调用者 |
| 排错（why-error） | 先提取错误码，再提取符号名（两阶段） | 合并两阶段结果 |
| 影响（what-impact） | 符号名 + usage/reference | BFS 展开全部调用者（深度2） |

**与现有技术的区别：**
现有方案对所有类型问题采用统一检索策略。本特征是领域内首个面向代码问答的差异化检索方案。

**技术效果：**
不同类型问题获得适配的检索策略，避免统一策略在不同意图间的折中损失。

### 从属增强特征 6：调用图传播重打分

**核心内容：**

在上述独立保护点的基础上，进一步在混合检索（FTS5 + 向量 + RRF 融合）之后，沿代码调用关系图的边对 Top-N 结果进行分数传播——每个种子的分数按衰减系数向直接调用者/被调用者传播，向二级节点衰减传播。

**与现有技术的区别：**
现有排序方法仅依赖文本匹配度或语义相似度。本特征引入代码特有的调用关系信号——被高频调用的核心函数即使在文本上匹配度略低，也能通过调用图传播获得更高的排序位置。同时设计了阶段式耦合结构（插入在 RRF 融合之后、LLM 上下文构建之前，限定传播种子数量为 Top-N），并通过双向传播（同时向调用者和被调用者传播）适配代码问答的差异化需求。

**技术效果：**
- 核心库函数排位提升，噪声结果排位下降
- 利用代码拓扑结构信号，无需额外训练
- 完全基于已有调用关系图，增量更新即可

---

### 从属增强特征 7：跨仓库动态范围限定

**核心内容：**

在上述独立保护点的基础上，进一步支持：通过 LLM 对问题进行搜索范围分类（单库/跨库对比/跨库调用链/全局搜索），当问题文本显式提到仓库名时自动限定范围，同时支持 `@repoName` 和 `@cross` 显式语法。

**与现有技术的区别：**
Sourcegraph Cody 的跨仓库搜索需企业级部署，Cursor/Aider 为单仓库设计。本特征在无需企业级基础设施的情况下，通过自然语言理解自动判定搜索范围。

**技术效果：**
多仓库场景下搜索范围自动判定，结果自动标注来源，无需用户手动选择。

## 八、具体实施方式

### 8.1 系统架构概述

本发明的实施系统按功能划分为五个层次，各层之间通过标准化的接口通信：

- **数据持久层**：负责代码调用关系图、Wiki 内容、问答知识库和待审区的结构化存储，支持高效的图遍历查询和全文检索
- **索引与分析层**：负责从源代码中提取代码结构信息、构建语义图谱，并提供检索、上下文分析、影响分析等能力（该层仅关注代码物理结构，与人工定义的 Wiki 业务模块层独立）
- **人工配置管理层**：提供业务模块定义、代码目录映射配置、角色权限管理、审核操作界面
- **问答与沉淀层**：负责意图分类、检索策略选择、调用图重打分、LLM 上下文构建和回答生成，以及自增编号分配、关联链接、版本化校准和待审管理
- **呈现层**：负责 Web 界面呈现、Wiki 知识库浏览（含公共问答池展示）、SSE 流式问答交互

系统支持多仓库部署——每个仓库独立构建语义图谱，问答时根据问题意图动态决定搜索范围。各层均支持水平扩展（多仓库并行索引、多实例负载均衡）。

以下为一种具体的组件选型和部署配置。

### 8.2 组件选型与部署配置

- **数据层**：使用 SQLite 数据库，包含三套独立的数据库——codegraph 索引库（存储代码符号、调用关系图）、qa 知识库（存储问答条目、校准记录、关联链接）、wiki 内容库（存储 Wiki 模块结构、内容版本、待审记录）
- **服务层**：Express HTTP 服务器，提供 SSE 流式问答端点、RESTful API 以及 Wiki 模块管理和审核接口
- **索引引擎**：基于 Tree-sitter 的 codegraph 引擎，提供多通道 FTS5 搜索和调用关系图查询
- **嵌入引擎**：通过 `@xenova/transformers` 加载本地 ONNX 模型（all-MiniLM-L6-v2，384 维），为代码符号和查询文本生成嵌入向量
- **LLM 推理**：兼容 OpenAI、DeepSeek、Anthropic 等主流 LLM API 接口
- **可选重排序引擎**：为提升最终排序质量，可配置一个独立的交叉编码器重排序服务，对 RRF 融合结果进行二次精排——该服务以子进程或独立容器方式部署，与主服务通过标准输入输出通信

### 8.3 问答与知识沉淀整体流程

本系统的完整知识沉淀流程包含五个阶段：

```
阶段一：人工配置业务骨架
  ├─ 管理员定义 Wiki 业务模块
  ├─ 配置各模块的代码目录映射
  └─ 确认骨架结构

阶段二：AI 按映射生成模块文档
  ├─ AI 读取映射表
  ├─ 扫描绑定目录中的代码
  └─ 自动生成模块底层 Wiki 子文档

阶段三：问答交互（增量采集）
  ├─ 用户输入自然语言问题
  ├─ 分配自增编号，检测引用语法，建立关联链接
  ├─ 意图分析 → 差异化检索 → 调用图重打分
  ├─ LLM 生成回答（锚定 Wiki 引用）
  └─ 自动入库，关联至对应业务模块待审区

阶段四：待审缓冲与人工审核
  ├─ AI 产出进入对应模块待审核待审区
  ├─ 管理员查看待审条目及差异对比
  └─ 执行：通过 / 修改后通过 / 驳回

阶段五：回流更新
  ├─ 更新正式 Wiki 模块内容
  ├─ 刷新向量检索索引
  └─ 更新后的 Wiki 作为后续问答的统一基准
```

单次问答处理的详细子流程：

```
步骤 1：用户输入自然语言问题
步骤 2：分配自增编号，检测引用语法，建立双向关联链接
步骤 3：意图分析（LLM 主分类 + 规则降级）→ 6 种意图之一 + 搜索范围分类
步骤 4：按意图选择检索策略，构造差异化检索词
步骤 5：混合搜索执行（FTS5 + 向量搜索 + RRF 融合 + 调用图传播重打分）
步骤 6：构建 LLM 上下文 + 选择回答模板（intent × domain）
步骤 7：SSE 流式 LLM 生成回答（含 Wiki 锚定引用）
步骤 8：自动入库沉淀至对应业务模块的待审区
```

### 8.4 关键参数

在本实施方式中：

- 自增编号起始值：10000
- RRF 融合参数 k：60
- 调用图传播衰减系数：0.15
- 调用图传播种子数（Top-N）：3
- 向量嵌入维度：384（all-MiniLM-L6-v2）
- 向量搜索方式：暴力余弦相似度（内存加载）
- LLM 温度参数：0（意图分类）/ 0.1（回答生成）

### 8.5 应用场景

本系统适用于以下场景：

1. **团队级多仓库代码知识库**：团队在多个仓库上工作，通过问答交互逐步沉淀每个仓库的代码知识
2. **新成员入职培训**：新成员可通过问答快速了解仓库架构和关键模块
3. **代码评审辅助**：评审人员通过问答了解受影响的代码范围和调用链
4. **故障排查**：开发人员上传错误日志，系统自动分析定位问题根因
5. **知识交接**：通过自增编号引用和双向关联链，团队成员可追溯问题的完整上下文

## 九、摘要

本发明公开了一种人机协同的代码知识 Wiki 迭代闭环沉淀方法及系统。该方法通过人工与 AI 权责分层的协同架构，由人工定义 Wiki 业务模块结构和代码目录映射，AI 按映射扫描代码自动生成底层模块文档；通过问答、批注等多渠道采集增量知识，经自增编号、双向关联和版本化校准沉淀为结构化知识资产，并经由待审缓冲池隔离和人工审核终审后回流至正式 Wiki 模块，形成"人工定骨架→AI 填充→增量采集→审核→回流复用"的完整知识闭环。在此基础上，还进一步包括意图自适应检索、调用图传播重打分和跨仓库动态范围限定等增强手段。本发明将人机协同与知识闭环沉淀相结合，使团队代码知识从一次性消耗品转变为人工可控、可积累、可复用的结构化知识资产。
